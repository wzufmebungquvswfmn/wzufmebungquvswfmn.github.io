utf8

UTF-8 错误

在 UTF-8 的规则里，字节被分成几种角色：

  • 普通人 (ASCII)： 0 到 127。自己就能成事。
  • 队长 (起始字节)： 192 到 244。它告诉解析器：“我是队长，后面跟着 1 到 3 个跟班”。其开头若有 n 个 1，就是 n - 1 字节队长，如：1110xxxx 后面需跟2个跟班。
  • 跟班 (连续字节)： 128 到 191，即二进制10开头。它必须跟在队长后面，不能单独出现。

根据以上规则，utf8的错误有以下几种：

  • 跟班前面不是队长：
    如 b00000000,b10000000 ，错误长度为 1
  • 队长后面跟的跟班数量不对：
    如 b11100000,b10000000,b00000000，本应跟两个跟班但是只跟了一个，错误长度为 2

内存对齐

给定一个地址 addr 和一个对齐 align
任务是把 addr 向上或者向下与 align 对齐，例如：

addr = 3， align = 8, then:  
align_up_addr = 8, align_down_addr

向下对齐即末尾置0， 即 align_down_addr = addr & !(align - 1)
向上对齐需检查是否需要进位，然后在末尾置0，即 align_up_addr = (addr + align - 1) & !(align - 1)

Clone

对于没有实现 Copy 的类型，若想达到 Copy 的效果，那就使用 Clone，但是对于实现了的类型 Copy 等价于 Clone

Clone传入的参数是不可变借用：

fn main() {
    let s1 = String::from("hello");
    let s2 = s1.clone(); // 等价于 let s2 = s1.clone(&s1); 这显然是更清晰的写法
    println!("s1: {}, s2: {}", s1, s2);
}

引用的互斥

一个数据只能有一个可变引用，或者多个不可变引用，即一个可变引用与其它引用互斥, 在可变引用存在时不允许其它引用存在。

对所有者自己而言，情况则很有意思，当一个数据被声明为 mut 时，在存在其它引用期间，你可以根据行为来将所有者视为以下三种状态：

  • 无视所有者： 当所有者在这期间什么行为都没有，所有者不被视作引用
  • 不可变引用： 当所有者在这期间只读取数据，它会被当做不可变引用
  • 可变引用： 当所有者在这期间修改数据，它会被当做可变引用，此时其它引用实际上是不允许存在的，否则会报错

mut 标识

一个数据有一个所有者，mut 标识的作用是让我们能够通过所有者修改这块数据，不使用 mut 只是不能通过这个所有者修改数据，但不代表数据本身不可变，事实上既然数据在那，那么如果愿意的话，总有办法修改它的，看以下例子：

#![allow(unused)]
fn main() {
let v = vec![1, 2, 3];
for mut x in v {
    x += 1;
    print!("{} ", x);
}
}

输出为：

2 3 4

v 被声明为不可变，但循环中仍能修改并打印，因为 for-in 结构实际上对容器进行迭代，而不加 & 相当于调用 v.into_iter()，它让 v 失去对这块内存的所有权而生成一个迭代器，迭代器里的元素的所有权在迭代过程中被转移给 x，x 又被声明为 mut，可以对每个元素进行修改。

Rust 结构体

Rust 常见有三种结构体：普通结构体、元组结构体、单元结构体。下面依次补全定义、创建实例和常用操作。 Rust 为结构体实现了自动解引用，当一个结构体被引用时，为了访问它的成员或方法，无需像C语言那样 p->member,只需使用p.member，它等价于(*p).member

1. 普通结构体（Regular Struct）

定义

#![allow(unused)]
fn main() {
struct ColorRegularStruct {
    red: u8,
    green: u8,
    blue: u8,
}
}

创建实例

#![allow(unused)]
fn main() {
let green = ColorRegularStruct {
    red: 0,
    green: 255,
    blue: 0,
};
}

访问字段

#![allow(unused)]
fn main() {
let g = green.green;
}

可变实例与修改字段

#![allow(unused)]
fn main() {
let mut color = ColorRegularStruct {
    red: 255,
    green: 255,
    blue: 255,
};
color.red = 0;
}

结构体更新语法

#![allow(unused)]
fn main() {
let base = ColorRegularStruct {
    red: 10,
    green: 20,
    blue: 30,
};

let another = ColorRegularStruct {
    red: 0,
    ..base
};
}

注意：使用 ..base 之后，base 中被“移动”的字段将不可再用（取决于字段类型是否实现 Copy）。

2. 元组结构体（Tuple Struct）

定义

#![allow(unused)]
fn main() {
struct ColorTupleStruct(u8, u8, u8);
}

创建实例

#![allow(unused)]
fn main() {
let green = ColorTupleStruct(0, 255, 0);
}

访问字段

#![allow(unused)]
fn main() {
let g = green.1;
}

3. 单元结构体（Unit Struct / 空结构体）

定义

#![allow(unused)]
fn main() {
struct UnitStruct;
}

创建实例

#![allow(unused)]
fn main() {
let unit = UnitStruct;
}

常见用途

  • 作为“类型标记”（type marker）
  • 实现 trait 而不需要存储数据

4. 结构体方法与关联函数（补充）

定义方法

#![allow(unused)]
fn main() {
impl ColorRegularStruct {
    fn is_green(&self) -> bool {
        self.red == 0 && self.green == 255 && self.blue == 0
    }
}
}

关联函数（类似静态方法）

#![allow(unused)]
fn main() {
impl ColorRegularStruct {
    fn green() -> Self {
        Self { red: 0, green: 255, blue: 0 }
    }
}
}

使用

#![allow(unused)]
fn main() {
let c = ColorRegularStruct::green();
let ok = c.is_green();
}

字符串比较

rust 中为String类型实现了 PartialEq 和 Eq 两个trait，所以可以直接使用==、!=来进行比较；切片、字面量都可以用来比较

如果想要比较两个String的地址，可以使用std::ptr::eq(&s1, &s2) 或 转换成内存地址指针：

#![allow(unused)]
fn main() {
let s1 = String::from("hello");
let s2 = String::from("hello");

let s1_ptr = &s1 as *const String;
let s2_ptr = &s2 as *const String;

assert!(s1_ptr != s2_ptr);
}

字符串相关转换

从 Vec::<u8> 到 String 的转换

  • String::from_utf8(vec: [u8]):
    · 会夺走vec的所有权，
    · 伴随着严格的错误处理，不允许无效u8字符，返回Result<String, FromUtf8Error>,
    · 其中 FromUtf8Error 结构如下：

    FromUtf8Error { 
    bytes, // 存储原有的数据，使其不至于丢失 
    error: Utf8Error { 
        valid_up_to: usize，  // 第一个无效字符的索引，也是最长的有效字符前缀的长度
        error_len: Option<u8> // 无效字符的长度，详见 utf8 章节
    } 
}

  • String::from_utf8_lossy(&vec: &[u8]):
    · 只拿vec的借用，无条件转换，返回 Cow<'_, str>
    · 若是全都有效字符，则返回 Cow::Borrowed，直接将原数组作为String， 无额外的拷贝与内存分配
    · 若存在无效字符，则返回 Cow::Owned，将有效字符拷贝到新分配的内存中，无效字符用一个空字符 U+FFFD 代替

  • String::from_utf8_unchecked(vec: [u8]):
    · 会夺走vec的所有权，
    · 无条件转换，返回 String
    · 无错误处理，若存在无效字符，会导致程序崩溃，但是很快

以上[u8]均可替换为Vec或&[u8]，根据方法的参数类型而定

从 String 到 &[u8] 的转换

  • as_bytes 方法返回字符串的字节切片 &[u8]，这是一个安全的操作，因为字符串在 Rust 中是 UTF-8 编码的，每个字符都由一个或多个字节组成，所以直接获取字节切片是没有问题的。

rust 中的标准输入

在 rust 中，标准输入可以通过 std::io 模块中的 stdin 函数来获取。这个函数返回一个 Stdin 结构体的实例，可以用来读取用户输入的数据。

rust 的输入哲学

rust 倾向于一次性读取所有输入，而不是逐行读取，这样可以更高效地处理输入数据。下面是一些标准示例，展示如何使用 stdin 来读取用户输入：

  • 一次性读入：
use std::io::{self, Read};

fn main() {
    let mut input = String::new();
    let stdin = io::stdin();
    stdin.read_to_string(&mut input).unwrap();
    // 按分隔符（包括空格、制表符、换行符等）划分输入：
    let mut tokens = input.split_whitespace();
}

对于上面得到的tokens，可以使用 next() 方法来逐个获取输入的值，并进行类型转换：

#![allow(unused)]
fn main() {
// 读取为字符串：
let s: String = tokens.next().unwrap().to_string();
// 读取为数字类型：
let n: i32 = tokens.next().unwrap().parse().unwrap();
// 读取为数组(第一项为数组大小n，后续项为数组元素)：
let n: usize = tokens.next().unwrap().parse().unwrap();
let arr: Vec<i32> = tokens.take(n).filter_map(|s| s.parse().ok()).collect(); // 或者写成 let arr: Vec<i32> = (0..n).map(|_| tokens.next().unwrap().parse().unwrap()).collect(); 也具有一样的效果
}

若是觉得.next().unwrap().parse().unwrap()太麻烦，可以写成宏：

#![allow(unused)]
fn main() {
use std::io::{self, Read};
macro_rules! read {
    ($expr:expr, $t:ty) => {
        $expr.next().unwrap().parse::<$t>().unwrap()
    };
    ($expr:expr) => {
        $expr.next().unwrap().to_string()
};  
}
}
#![allow(unused)]
fn main() {
// 使用宏读取输入：
let s: String = read!(tokens);
let n: i32 = read!(tokens, i32);
let n: usize = read!(tokens, usize);
let arr: Vec<i32> = (0..n).map(|_| read!(tokens, i32)).collect();
}

    }

}

Read 和 BufRead

在 Rust 中，Read 和 BufRead 是两个重要的 trait，用于处理输入输出操作。以下是二者的区别：

  • Read trait：提供了基本的读取方法，适用于一次性读取大量数据。在实际使用中，stdin 返回的 Stdin 结构体实现了 Read trait，因此你可以直接使用 read_to_string（读取到一个字符串）、read（读取到字节数组）方法来一次性读取所有输入。
  • BufRead trait：提供了缓冲读取的方法，适用于逐行读取输入，可以提高效率并减少系统调用的次数。如 stdin.lock() 获取一个实现了 BufRead trait 的锁定版本，从而使用 lines（读取所有行）、read_line（读取单行）方法来逐行读取输入。

集合数据结构

Rust 标准库文档的 std::collections 模块中这样写到：

you should probably just use Vec or HashMap. These two collections cover most use cases for generic data storage and processing.

也就是说绝大多数场景里，Vec 和 HashMap 已经能够解决问题。但无论如何，Rust 标准库还是提供了多种集合数据结构来满足不同的需求：

  • Vec
  • HashMap/HashSet
  • BinaryHeap
  • VecDeque
  • BTreeMap/BTreeSet

如何选择集合

Vec、HashMap、HashSet 自不必多说。

双向队列有时也会有用，尤其是 BFS 中从队头取元素、从队尾加入元素，使用 VecDeque。

如果需要每次取出最大值或最小值，使用 BinaryHeap 作为优先队列，它提供了性能极好的有序插入。

如果需要 key 保持有序，或者需要范围查询，使用 BTreeMap / BTreeSet。

添加和删除

  • push 和 pop 用于在尾部添加和删除元素，是最常见的栈操作

  • insert 可以在指定位置插入元素，remove 可以删除指定位置的元素：

#![allow(unused)]
fn main() {
let mut nums = vec![1, 3];

nums.insert(1, 2);
assert_eq!(nums, vec![1, 2, 3]);

let x = nums.remove(1);
assert_eq!(x, 2);
assert_eq!(nums, vec![1, 3]);
}

需要注意，insert 和 remove 会移动后面的元素，时间复杂度是 O(n)。

如果不关心元素顺序，可以使用 swap_remove，它会把最后一个元素换到被删除的位置，因此更快：

#![allow(unused)]
fn main() {
let mut nums = vec![10, 20, 30, 40];

let x = nums.swap_remove(1);

assert_eq!(x, 20);
assert_eq!(nums, vec![10, 40, 30]);
}

访问元素

  • 使用下标访问时，如果越界会 panic

  • 如果不确定下标是否合法，使用 get 更安全：

#![allow(unused)]
fn main() {
let nums = vec![1, 2, 3];

if let Some(x) = nums.get(10) {
    println!("{}", x);
}
}
  • 只需首尾访问方法：
#![allow(unused)]
fn main() {
let nums = vec![1, 2, 3];

assert_eq!(nums.first(), Some(&1));
assert_eq!(nums.last(), Some(&3));
}

如果需要修改元素，可以使用 get_mut获取元素的可变引用：

#![allow(unused)]
fn main() {
let mut nums = vec![1, 2, 3];

if let Some(x) = nums.get_mut(0) {
    *x = 10;
}

assert_eq!(nums, vec![10, 2, 3]);
}

原地排序

  • sort 默认升序排序：

  • 降序可以使用 sort_by：

#![allow(unused)]
fn main() {
let mut nums = vec![3, 1, 2];

nums.sort_by(|a, b| b.cmp(a));

assert_eq!(nums, vec![3, 2, 1]);
}
  • 如果按某个字段或计算结果排序，使用 sort_by_key：
#![allow(unused)]
fn main() {
let mut words = vec!["aaa", "b", "cc"];

words.sort_by_key(|s| s.len());

assert_eq!(words, vec!["b", "cc", "aaa"]);
}
  • reverse 会直接翻转整个 Vec：
#![allow(unused)]
fn main() {
let mut nums = vec![1, 2, 3];

nums.reverse();

assert_eq!(nums, vec![3, 2, 1]);
}

原地去重

  • dedup 只会删除相邻重复元素，所以常见用法是先排序，再去重：
#![allow(unused)]
fn main() {
let mut nums = vec![3, 1, 2, 3, 2];

nums.sort();
nums.dedup();

assert_eq!(nums, vec![1, 2, 3]);
}

原地过滤 retain

  • retain 会保留满足条件的元素，相当于原地过滤：
#![allow(unused)]
fn main() {
let mut nums = vec![1, 2, 3, 4, 5];

nums.retain(|x| *x % 2 == 1);

assert_eq!(nums, vec![1, 3, 5]);
}

查找

  • contains 用于判断元素是否存在：
#![allow(unused)]
fn main() {
let nums = vec![1, 2, 3];

assert!(nums.contains(&2));
}
  • 如果 Vec 已经有序，可以使用 binary_search 二分查找：
#![allow(unused)]
fn main() {
let nums = vec![1, 3, 5, 7];

assert_eq!(nums.binary_search(&5), Ok(2));
assert_eq!(nums.binary_search(&4), Err(2));
}

Ok(index) 表示找到了元素，Err(index) 表示没找到，并给出可以插入的位置。

切片相关方法

很多 Vec 方法其实来自切片 [T]，所以数组、切片、Vec 都能用。

  • windows 用于生成固定长度的滑动窗口：
#![allow(unused)]
fn main() {
let nums = vec![1, 2, 3, 4];

for w in nums.windows(2) {
    println!("{:?}", w);
}
}

输出的窗口依次是 [1, 2]、[2, 3]、[3, 4]。

  • chunks 用于按固定大小分组：
#![allow(unused)]
fn main() {
let nums = vec![1, 2, 3, 4, 5];

for chunk in nums.chunks(2) {
    println!("{:?}", chunk);
}
}

输出的分组是 [1, 2]、[3, 4]、[5]。

  • split_at 可以把切片从某个位置分成两段：
#![allow(unused)]
fn main() {
let nums = vec![1, 2, 3, 4];

let (left, right) = nums.split_at(2);

assert_eq!(left, &[1, 2]);
assert_eq!(right, &[3, 4]);
}

    }
}
}

访问

  • get 通过 key 获取 value:
#![allow(unused)]
fn main() {

    println!("{}", count);
}
}

#![allow(unused)]
fn main() {

查询

  • contains_key 判断 key 是否存在,返回 true 或 false：
#![allow(unused)]

}

删除

  • remove 通过 key 删除元素,并获取旧值：
#![allow(unused)]
fn main() {

}

遍历键值对

  • values 返回对 value 的不可变引用迭代器

  • into_values 获取 value 的所有权迭代器，同时拿走 HashMap 的所有权

#![allow(unused)]

BinaryHeap

BinaryHeap<T> 是 Rust 标准库中的二叉堆，可以当作优先队列使用。它默认是大顶堆，也就是说每次 pop 都会弹出当前最大的元素。

它的比较标准是根据字段顺序来的，所以元组或结构体（需要实现 Ord trait）的字段顺序也会影响比较结果。

注意:

BinaryHeap 只能保证堆顶是最大值，并不保证整体有序，如果想得到有序结果，需要不断 pop。

大顶堆

#![allow(unused)]

use std::collections::BinaryHeap;

let mut heap = BinaryHeap::new();

heap.push(2);
heap.push(8);

assert_eq!(heap.pop(), Some(8));

}

可以使用 std::cmp::Reverse 把排序规则反过来，从而得到小顶堆：

use std::collections::BinaryHeap;

heap.push(Reverse(2));
heap.push(Reverse(8));

assert_eq!(heap.pop(), Some(Reverse(5)));
assert_eq!(heap.pop(), Some(Reverse(8)));
}

Top K 模板

如果要找最大的 k 个元素，可以直接用大顶堆全部加入后弹出 k 次。

#![allow(unused)]
fn main() {
use std::collections::BinaryHeap;

let nums = vec![3, 1, 5, 2, 4];
let mut heap = BinaryHeap::from(nums);

let mut ans = Vec::new();
for _ in 0..3 {
    if let Some(x) = heap.pop() {
        ans.push(x);
    }
}

assert_eq!(ans, vec![5, 4, 3]);
}

如果数据很多，只想维护最大的 k 个元素，也可以用小顶堆。堆中只保留 k 个元素，堆顶就是当前 top k 中最小的那个：

#![allow(unused)]
fn main() {
use std::cmp::Reverse;
use std::collections::BinaryHeap;

let nums = vec![3, 1, 5, 2, 4];
let k = 3;
let mut heap = BinaryHeap::new();

for x in nums {
    heap.push(Reverse(x));
    if heap.len() > k {
        heap.pop();
    }
}

let mut ans: Vec<_> = heap.into_iter().map(|Reverse(x)| x).collect();
ans.sort();

assert_eq!(ans, vec![3, 4, 5]);
}

VecDeque

VecDeque<T> 是双端队列，可以在队头和队尾高效地添加、删除元素。最常见的用途是 BFS 队列，也可以用于滑动窗口。

作为队列使用

普通队列是先进先出，通常从队尾加入，从队头弹出：

#![allow(unused)]
fn main() {
use std::collections::VecDeque;

let mut queue = VecDeque::new();

queue.push_back(1);
queue.push_back(2);
queue.push_back(3);

assert_eq!(queue.pop_front(), Some(1));
assert_eq!(queue.pop_front(), Some(2));
assert_eq!(queue.pop_front(), Some(3));
assert_eq!(queue.pop_front(), None);
}

BFS 中经常这样写：

#![allow(unused)]
fn main() {
while !queue.is_empty() {
    let (r, c) = queue.pop_front().unwrap();
    // 处理 (r, c) 与其邻居
}
// 或者直接写成
while let Some((r, c)) = queue.pop_front() {
    // 处理 (r, c) 与其邻居
}
}

作为双端队列使用

VecDeque 两端都可以操作：

#![allow(unused)]

use std::collections::VecDeque;

let mut deque = VecDeque::new();

deque.push_front(1);
deque.push_back(2);

assert_eq!(deque.front(), Some(&1));
assert_eq!(deque.back(), Some(&2));

assert_eq!(deque.pop_back(), Some(2));
assert_eq!(deque.pop_front(), Some(1));
}

单调队列

求滑动窗口最大值，经常会使用单调队列。队列里存下标，并让对应的值保持单调。

下面是维护窗口最大值的常见写法：

#![allow(unused)]
fn main() {
use std::collections::VecDeque;

let nums = vec![1, 3, -1, -3, 5, 3, 6, 7];
let k = 3; // 窗口大小
let mut deque: VecDeque<usize> = VecDeque::new();
let mut ans = Vec::new();

for i in 0..nums.len() {
    while let Some(&j) = deque.back() {
        if nums[j] <= nums[i] {
            deque.pop_back();
        } else {
            break;
        }
    }

    deque.push_back(i);

    if let Some(&j) = deque.front() {
        if j + k <= i {
            deque.pop_front();
        }
    }

    if i + 1 >= k {
        ans.push(nums[*deque.front().unwrap()]);
    }
}

assert_eq!(ans, vec![3, 3, 5, 5, 6, 7]);
}

这里的核心思想是：队头始终保存当前窗口最大值的下标。

let mut map = BTreeMap::new();

map.insert(3, "c");

#![allow(unused)]

BTreeSet<T> 可以看作有序的 HashSet，常用于需要自动排序、去重、范围查询的场景。

基本使用

#![allow(unused)]

泛型(Generic) 和特征 (Trait)

记录一些零碎的泛型和特征的知识点

泛 型

泛型是一种类型参数，可以是任何类型，在编译时会被替换为具体的类型

  • fn func<T>(arg: T) -> N;
  • struct S<T> { arg: T, .. };
  • enum E<T> { A(T), B, .. };
  • impl<T: 约束条件> S<T> { fn func<T>(arg: T) -> N { ..T受约束的调用.. } }
    • 约束条件 为 特征 或 类型 或 特征+类型 或 特征+特征 等
    • 只有< >里面和 where 子句等处的T才能跟约束条件，函数列表中的类型参数一般跟普通类型、泛型或具体实现，如 i32, T, impl Trait 等

特征

  • trait TraitName { fn func(&self) -> N; }
  • impl TraitName for S { fn func(&self) -> N { .. } }
  • fn func(arg: impl TraitName) <==> fn func<T: TraitName>(arg: T)
  • fn func(arg: impl TraitName + TraitName2) <==> fn func<T: TraitName + TraitName2>(arg: T) //多重约束

Move、Copy 和 Clone

简介

  • Move 是默认行为，在栈上进行位拷贝（实际可能不进行拷贝），所有权转移，原变量失效。
  • Copy 也在栈上进行位拷贝，原变量仍然有效。适用于完全在栈上的类型，包括固定大小的数组。它与 Drop 不兼容。
  • Clone 为无法实现 Copy 的类型提供深拷贝，若不采用引用计数会比较昂贵。

派生

  • 以下是常用的派生方式，适用于简单类型组合：
    #[derive(Copy, Clone)]
  • 这样写的原因在于 Copy 继承自 Clone（Copy: Clone），所以必须同时实现 Clone。

Sized 、!Sized 和 ?Sized

  • Sized 标记编译器已知大小固定的类型，默认所有类型都满足 Sized 约束。
  • !Sized 表示类型大小在编译时未知，如动态大小类型 str 和 dyn Trait。
  • ?Sized 表示类型可能是 Sized 也可能是 !Sized，所以需要作为动态大小类型 (DST) 处理，即传参时传引用，因为地址长度总是固定的 8 字节或 16 字节，例如：
#![allow(unused)]
fn main() {
fn print_it<T: ?Sized>(arg: &T) {
    // ...
}
}

Send 和 Sync

这两个特征标记类型是否可以安全的在线程之间传递共享，是典型的标记特征，无具体实现。多线程并发 中有提到。

Send

它标记一个类型可以在线程间安全的转移所有权，即可以把它 move 到其它线程里去。

大多数 Rust 类型都是 Send 的；而典型的非 Send 类型是 Rc<T>，因为它的引用计数器不是线程安全的，还有裸指针 *const T 和 *mut T，因为它们没有任何所有权和生命周期的保证。

Sync

标记一个类型可以在多个线程间安全的共享引用，即 T: Sync <=> &T: Send。
典型非 Sync 的类型是 Cell<T> 和 RefCell<T>，因为它们提供了内部可变性，但没有线程安全的保证。

线程安全组合：

T: Send => Mutex<T>: Sync 和 RwLock<T>: Sync T: Send + Sync => Arc<T>: Send + Sync

实现或派生

自动派生，但如果有裸指针或 Rc<T>，在确保线程安全的情况下，可以手动在unsafe块中实现

std::io 的核心： Read 和 Write 以及 BufRead 和 Seek

Read

实现了 Read 的读者，有必需（实现的）方法 read()：

#![allow(unused)]
fn main() {
// 从 self 读取数据到 buf 中，尽可能读满并返回实际读取的字节数
fn read(&mut self, buf: &mut [u8]) -> Result<usize>;
}

除此之外还有以下常用默认方法，它们都会尽可能读至self的EOF，并返回实际读取的字节数：

#![allow(unused)]
fn main() {
// 从 self 读取数据直到 EOF，追加到 buf 中
read_to_end(&mut self, buf: &mut Vec<u8>) -> Result<usize>  
// 从 self 读取数据直到 EOF，追加到 buf 中，要求字符是 UTF-8 编码
read_to_string(&mut self, buf: &mut String) -> Result<usize>
}

Write

实现了 Write 的写者，有必需方法 write() 和 flush()：

#![allow(unused)]
fn main() {
// 从 buf 中写入数据到 self 中，返回实际写入的字节数
fn write(&mut self, buf: &[u8]) -> Result<usize>;
// 把内部缓冲区（self的一个成员）的数据强制写入底层设备，确保数据被发送或保存
fn flush(&mut self) -> Result<()>;
}

Seek

它对读者或写者进行定位，有必需方法 seek()，配合枚举 SeekFrom 使用：

#![allow(unused)]
fn main() {
enum SeekFrom {
    Start(u64), // 从起始位置偏移
    End(i64),   // 从末尾位置偏移
    Current(i64) // 从当前位置偏移
}
// 将 self 的当前位置设置为 pos 所指定的位置，并返回新的位置
fn seek(&mut self, pos: SeekFrom) -> Result<u64>;
}

BufRead

带缓冲区的读者，继承自 Read，有必需方法 fill_buf() 和 consume()：

#![allow(unused)]

fn fill_buf(&mut self) -> Result<&[u8]>; // 返回内部缓冲区中可用的数据
fn consume(&mut self, amount: usize); // 从内部缓冲区中消费 amount 字节的数据
}

常有默认方法有：

fn main() {
// 读取一行 UTF-8 字符到 buf 中，返回实际读取的字节数
fn read_line(&mut self, buf: &mut String) -> Result<usize>;
// 返回一个迭代器，逐行读取数据
fn lines(self) -> Lines<Self> where Self: Sized;
}

BufReader::with_capacity(capacity: usize, inner: R) -> BufReader<R>```
}

来实现转换，其中 R: Read，capacity 是缓冲区的大小，默认为 8KB。

特别地，对 stdin 来说，lock() 方法可以将其转换为 BufReader,详见标准输入

PartialEq 和 Eq

  • PartialEq 用于比较两个值是否相等，提供了 == 和 != 操作符的实现，之所以叫 Partial，是因为浮点数中NaN不等于任何值，包括它自己。

PartialOrd 和 Ord

PartialOrd

  • 必需方法:
#![allow(unused)]
fn main() {
fn partial_cmp(&self, other: &Rhs) -> Option<Ordering>; // 实现和任何类型右值Rhs的比较
}

附赠了 lt（<）、le（<=）、gt（>）、ge（>=）四个方法的实现

  • Ordering 是一个枚举，表示比较结果：
#![allow(unused)]
fn main() {
enum Ordering {
    Less,    // 小于
    Equal,   // 等于
    Greater, // 大于
}
}

Ord： Eq + PartialOrd

  • 和 Eq 类似，必需方法:
#![allow(unused)]
fn main() {

}

提供max、min 以及 clamp等方法的实现:

#![allow(unused)]
fn main() {
// 将 self 限制在 min 和 max 之间
fn clamp(self, min: Self, max: Self) -> Self 
where Self: Sized + PartialOrd; 
}
  • #[derive(PartialOrd, Ord)] 可以派生, 比较按照字段在结构体中的顺序进行。

关联宏

From 和 Into

  • from 定义了如何从其它类型创建一个自己类型的值

  • into 恰恰相反，它定义了如何将自己类型转换为其它类型的值

只需实现 From 就会自动获得 Into 的实现

AsRef 和 AsMut

不同于 From 和 Into, 它们是不消耗所有权的转换:

#![allow(unused)]
fn main() {
fn as_ref(&self) -> &T;
fn as_mut(&mut self) -> &mut T;
}

这两个特征使得借用多态实现非常方便，如在函数入口加一个.as_ref()，就可以接受任何实现了 AsRef 的类型:

#![allow(unused)]
fn main() {
fn print_it(arg: impl AsRef<str>) {
    println!("{}", arg.as_ref());
}
}

Deref 和 DerefMut

Deref 和 DerefMut 是对智能指针解引用操作符 * 的重载，允许我们定义自己的智能指针类型，并使其行为类似于普通引用。

Deref 定义了如何通过 * 操作符获取智能指针内部的值的引用：

#![allow(unused)]
fn main() {
struct MyBox<T>(T);
impl<T> std::ops::Deref for MyBox<T> {
    type Target = T;
    fn deref(&self) -> &T {
        &self.0
    }
}
}

DerefMut 则是 Deref 的mut版本，继承自 Deref

除了显式使用 * 时会解引用，在函数传参与方法调用时也会自动解引用（解引用强制转换）。

Hash

std::hash 模块下的特征，实现了 Hash 类型可以变成一个哈希值，通过下方的必须方法：

#![allow(unused)]
fn main() {
fn hash<H: Hasher>(&self, state: &mut H); // 实现了 Hasher 的 H 是一个哈希生成器
}

实现通过 #[derive(Hash)] 可以派生，或者把需要计算的字段手动哈希：

#![allow(unused)]
fn main() {
fn hash<H: Hasher>(&self, state: &mut H) {
    self.field1.hash(state); // 只哈希 field1 字段

}

Borrow

borrow 模块有下面几个 trait，主要用来扩展借用和所有权的功能

  • Borrow 使得一个类型的引用能转换成另一个类型的引用，常见于 Hash 的键查找：
#![allow(unused)]
fn main() {
impl<Q: ?Sized> Borrow<Q> for K where K: Hash + Eq, Q: Hash + Eq {
    fn borrow(&self) -> &Q;
}
}

一个典型的例子是 String 实现了 Borrow<str>，使得对于 get(s: &str) 来说，传入的参数类型是 &str 和 &String 得到的哈希值相同。

  • BorrowMut 是 Borrow 的 mut 版本。
  • ToOwned 是 Clone 的泛用版本，提供了 to_owned() 方法，可以从一个引用类型直接创建一个拥有所有权的类型，例如 str 的 to_owned() 会返回一个 String。
  • Cow<'a,T>（Clone on Write, 写时克隆）是一个智能指针, 在需要写时（比如调用 to_mut()）就会用 to_owned() 克隆出一个拥有所有权的类型，否则保持对原始数据的引用。

}

闭包

本节简单介绍闭包概念和它的三类型

闭包

闭包类似于函数，不同之处在于，除了传入的参数之外，其还可以捕获环境中的变量：

#![allow(unused)]
fn main() {
let a = 12345; // 环境中的变量
let b = 54321; // 传入的参数
let func = |x|{println!("env_var: {a}\nparameter: {x}");};
func(b);
}
env_var: 12345
parameter: 54321

根据闭包对捕获变量的所有权的处理权限，可将闭包分为以下三类：

  • Fn:
    仅能对捕获的变量执行不可变借用，也就是只读，因而可以调用无限次
  • FnMut:
    可以对捕获的变量最大执行可变借用，也就是可读可写，也可以调用无限次
  • FnOnce:
    权力最大，可拿走捕获的变量的所有权，只能调用一次 要注意的几点是：
  1. 闭包分类的 唯一依据 是 闭包对捕获的变量所有权的权限大小
  2. 三种类型并不是并列关系，而是包含关系，从权限上来说Fn ⊂ FnMut ⊂ FnMut；因此一个Fn类型的闭包也是FnMut和FnOnce的
  3. 闭包的类型由编译器自动推断，遵循最小特权原则，也就是默认是最小不可变借用，若是必须采用Move而不是Copy，那么必须给予更大权限
  4. 闭包前添加move会将捕获的变量以函数参数同样的方式进行处理

智能指针

根源：Rust 所有权模型的矛盾

Rust 的核心设计是“所有权模型“：

  • 每个值只能有一个所有者
  • 所有权可以转移，但同一时刻只有一个所有者拥有值
  • 当所有者消失时，值自动释放（没有垃圾回收，没有手动 free）

好处：无需垃圾回收，内存安全得到编译期保证。
代价：数据流动受到限制。

问题：现实中很多数据结构需要“多个持有者“

在 C/Java 等语言中，你可以随意创建指针或引用，让多个对象指向同一个数据。但在 Rust 中，这违反了“唯一所有者“的规则。

例子：

#![allow(unused)]
fn main() {
let data = vec![1, 2, 3];
let ptr1 = &data;  // ✅ 借用
let ptr2 = &data;  // ✅ 借用

// 关键问题：ptr1 和 ptr2 都是"临时的"，只能在作用域内使用
// 如果我想跨越作用域持有它们呢？如果我想让 ptr1 或 ptr2 拥有 data 呢？
}

所有权模型的本质矛盾

所有权说“只有一个持有者“，但真实的数据结构经常需要多个持有者。

例如：

  • 二叉树：子节点被各自的父节点持有，但有时你也想在函数外持有某个子节点
  • 图：一个节点可能被多条边引用
  • 缓存/shared state：多个线程或多个组件共享同一份数据

智能指针：一个协议而非限制

Rust 没有放弃所有权，而是创造了“智能指针“来协商这个矛盾：

Rc<T> 的意思是：

“我不再坚持只有一个所有者。让这个值由引用计数管理，多个指针可以同时持有它。最后一个持有者消失时释放。”

这是一个权衡：

  • 失去了：编译期完全的所有权明确性
  • 获得了：能够实现真实的数据结构（树、图、共享状态）

总结：智能指针存在的根本原因是：Rust 的所有权模型太严格，无法表达某些真实的、合法的数据结构。智能指针通过运行时手段（引用计数、内部可变性）来突破这个限制。

新的矛盾：Rc 允许多个持有者，但为什么不让他们都能修改？

你可能会想：既然 Rc 已经允许了多个持有者，为什么还要限制它们只能读，不能写？

看这个场景：

#![allow(unused)]
fn main() {
let data = Rc::new(vec![1, 2, 3]);
let ptr1 = Rc::clone(&data);  // 持有者 1
let ptr2 = Rc::clone(&data);  // 持有者 2

// 如果允许这样:
ptr1[0] = 10;  // 持有者 1 正在修改
// 同时：
let x = ptr2[0];  // 持有者 2 正在读，读到的是什么？10 还是 1？
}

问题的根本：多个独立的持有者在不知道彼此存在的情况下同时访问数据。

为什么这很危险？

在有垃圾回收的语言中（Java/Python），你可以随意让多个引用修改同一个对象，GC 帮你管生命周期。但在 Rust 中，多个 Rc 持有者意味着：

  • 持有者 1 不知道持有者 2 何时会出现
  • 持有者 1 也不知道持有者 2 何时会消失
  • 如果允许修改，就无法保证“同一时刻只有一个人修改“

Rc 的安全保证：

  • 允许多个持有者 ✅
  • 通过引用计数管理生命周期 ✅
  • 但只能不可变访问 ← 这是为了保证安全

所以呢？RefCell 为什么敢允许修改？

#![allow(unused)]
fn main() {
let data = Rc::new(RefCell::new(vec![1, 2, 3]));
let ptr1 = Rc::clone(&data);
let ptr2 = Rc::clone(&data);

ptr1.borrow_mut()[0] = 10;  // 持有者 1 获取可变引用
// ptr2.borrow_mut()[0] = 20;  // ❌ 这会 panic！
}

RefCell 的安全保证：

  • 我不再依赖编译器来检查借用规则
  • 我在运行时检查：同一时刻最多一个可变引用
  • 如果你违反这个规则，我当场 panic

这是一个权衡：

  • Rc：信任编译器，但放弃修改能力
  • RefCell：放弃编译期检查，在运行时动态检查，获得修改能力

Rc<RefCell<T>> 的组合意思是：

“允许多个持有者（Rc），但每次修改时动态检查（RefCell），确保安全”

这也解释了为什么多线程用 Arc<Mutex<T>> 而不是 Arc<T>：

  • Arc = “多线程下的 Rc”
  • Mutex = “多线程下的 RefCell”（用锁代替运行时检查）

为什么多个修改者很危险？具体例子

“多个持有者不能随意修改“听起来很抽象。让我用一个 C 的例子（Rust 不允许你写，但 C 可以）来展示真实的危险：

例子 1：向量扩容导致悬垂指针

// 用 C 模拟 Rust 禁止的行为

int* data = malloc(sizeof(int) * 3);
data[0] = 1; data[1] = 2; data[2] = 3;

int* ptr1 = data;  // 指针 1
int* ptr2 = data;  // 指针 2

// 指针 1 向向量添加第 4 个元素
// 为了容纳 4 个元素，向量需要重新分配内存（假设原来容量只有 3）
realloc(data, sizeof(int) * 6);  // 可能返回新地址！

// 现在 data 指向新地址，但 ptr2 仍然指向旧地址
printf("%d", ptr2[0]);  // ❌ 悬垂指针！读取已释放内存或垃圾数据

在 Rust 中，如果允许这样：

#![allow(unused)]
fn main() {
let data = Rc::new(vec![1, 2, 3]);  // 容量为 3
let ptr1 = Rc::clone(&data);
let ptr2 = Rc::clone(&data);

ptr1[0] = 10;  // 没问题
ptr1.push(4);  // ❌ 向量扩容，可能重新分配内存地址！

// ptr2 现在引用的内存地址可能已经失效
println!("{}", ptr2[0]);  // ❌ 未定义行为（UndefinedBehavior）
}

例子 2：迭代器失效

#![allow(unused)]
fn main() {
let data = Rc::new(vec![1, 2, 3]);
let ptr1 = Rc::clone(&data);
let ptr2 = Rc::clone(&data);

let iter = ptr1.iter();  // 开始在 ptr1 上迭代

ptr2.push(4);  // ❌ 在迭代进行时修改底层数据！

for item in iter {
    println!("{}", item);  // ❌ 迭代器状态已失效，可能读垃圾数据
}
}

在 Java/Python 中，你会得到 ConcurrentModificationException。在 C++ 中，你会得到未定义行为。

例子 3：逻辑不一致

#![allow(unused)]
fn main() {
struct LinkedListNode {
    val: i32,
    next: Option<Rc<LinkedListNode>>,
}

let node1 = Rc::new(LinkedListNode { val: 1, next: None });
let node2 = Rc::clone(&node1);

// 如果允许修改：
node1.next = Some(Rc::new(LinkedListNode { val: 2, next: None }));

// node2 现在的下一个节点是什么？
// 它是最初的 None，还是现在的 Some(2)？
// 两个持有者对"链表结构"有不同的理解了！
}

问题的本质

当多个持有者各自独立修改同一个数据时：

持有者 A持有者 B结果
读数据读数据✅ 安全
读数据修改数据❌ 数据竞争：A 读到不一致的状态
修改数据修改数据❌ 数据竞争：整个数据结构破坏
修改数据（扩容）任何操作❌ 内存失效：悬垂指针、UB

Rust 的三个解决方案

1. Rc 的选择：禁止修改，只允许不可变借用

#![allow(unused)]
fn main() {
let data = Rc::new(vec![1, 2, 3]);
// 所有持有者都只能读，永远不会出现修改冲突
}

2. RefCell 的选择：允许修改，但运行时检查

#![allow(unused)]
fn main() {
let data = Rc::new(RefCell::new(vec![1, 2, 3]));
let ptr1 = Rc::clone(&data);

let mut borrow1 = ptr1.borrow_mut();  // 获得可变借用
// let mut borrow2 = ptr2.borrow_mut();  // ❌ 运行时 panic，因为 borrow1 还活着
}

3. Mutex 的选择（多线程）：允许修改，用锁同步

#![allow(unused)]
fn main() {
let data = Arc::new(Mutex::new(vec![1, 2, 3]));
let ptr1 = Arc::clone(&data);
let ptr2 = Arc::clone(&data);

let mut guard1 = ptr1.lock();  // 获得锁
// let mut guard2 = ptr2.lock();  // 会阻塞，直到 guard1 释放
}

总结三层逻辑

  1. 第一层：Rust 所有权太严格（只能一个持有者）

    • 解决：Rc 允许多个持有者
    • 代价：不能修改

  2. 第二层：但我需要修改！Rc 为什么不让改？

    • 原因：多个独立的修改者会导致数据竞争和内存不安全
    • 解决：RefCell 用运行时检查来确保安全修改
    • 代价：运行时开销 + 可能 panic

  3. 第三层：Rc<RefCell<T>> 的组合

    • 允许多个持有者（Rc）
    • 允许安全修改（RefCell 的运行时检查）
    • 这就是现实数据结构所需的完整解决方案

常见理解误区（澄清）

误区 1：“Rc 其实并不违背所有权机制，只是使用引用计数来看似拥有所有权”

正确理解：Rc 确实违背了 编译期的所有权保证，但用运行时的引用计数机制来补偿这种违背。

  • 所有权机制说：“编译期间，我能保证内存安全”
  • Rc 说：“我放弃编译期保证，改用运行时的引用计数来保证”

这是一个权衡，不是没有违背。代价是：

  • 失去了编译期的完全确定性
  • 得到了灵活性（多个持有者）
  • 多了运行时开销（引用计数）

误区 2：“为什么要叫’引用’计数？”

这个名字来自于：

  • 每个 Rc 都是对底层数据的一个引用
  • “引用计数“就是“有多少个 Rc 指向这个数据”
  • 与其说是“计算引用“，不如说是“计数多少个引用“
#![allow(unused)]
fn main() {
let data = Rc::new(5);
let ptr1 = Rc::clone(&data);  // 第 1 个引用
let ptr2 = Rc::clone(&data);  // 第 2 个引用
let ptr3 = Rc::clone(&data);  // 第 3 个引用
println!("{}", Rc::strong_count(&data));  // 输出: 3，共 3 个引用
}

误区 3：“RefCell 让数据无视所有权限制进行修改”

正确理解：RefCell 不是“无视“所有权限制，而是把编译期检查变成运行时检查。

  • Rust 的借用规则不变：“同一时刻最多一个可变引用”
  • 区别在于：
    • 普通的 &T 和 &mut T：编译器在编译期强制这个规则
    • RefCell<T>：允许你编译通过，但在运行时强制这个规则
#![allow(unused)]
fn main() {
let data = RefCell::new(5);

// 编译时允许这样写
let r1 = data.borrow();      // 不可变引用
let r2 = data.borrow();      // 又一个不可变引用
println!("{}", *r1);         // ✅ 可以

// 但这样会 panic
let r3 = data.borrow_mut();  // ❌ 运行时 panic：已经有不可变引用了

// drop(r1);  // 如果释放了 r1
// let r3 = data.borrow_mut();  // ✅ 现在可以了
}

所以 RefCell 的本质是：我相信你会遵守借用规则，但我会在运行时监视你。

最终总结（更精确的表述）

Rc 和 RefCell 是对 Rust 所有权机制的两个不同方向的扩展：

  1. Rc：突破“只能一个所有者“的限制

    • 允许多个持有者共享所有权
    • 代价：放弃修改能力（编译器保证不可变）
    • 运行时开销：引用计数

  2. RefCell：突破“可变引用的编译期检查“的限制

    • 允许在持有不可变引用时修改数据
    • 代价：把检查推迟到运行时，可能 panic
    • 运行时开销：借用状态追踪

  3. Rc<RefCell<T>>：同时突破两个限制

    • 允许多个持有者且都能修改
    • 代价：失去编译期安全保证，转而用运行时动态检查
    • 运行时开销：引用计数 + 借用状态追踪

核心理念：Rust 的所有权机制是“零成本的内存安全“，但某些数据结构（树、图、共享状态）无法用这个模式表达。Rc 和 RefCell 是说“我知道这会失去零成本和编译期保证，但我愿意用运行时手段来换取这个灵活性“。

如果你只需要一个清晰的认知来解决你的二叉树问题，就记住这一个表格：

一个所有者多个所有者
编译期检查T / &T / &mut T❌ 不存在
运行时检查RefCell<T>Rc<RefCell<T>>

你的二叉树问题需要：

  • 多个所有者：因为父节点持有子节点，你也想持有子节点 → 需要 Rc
  • 运行时检查：因为需要修改节点 → 需要 RefCell

所以答案就是 Rc<RefCell<T>>。

就这样。不用理解运行时如何工作，只需要知道“这个组合提供了我需要的能力“。

注：从最简单的例子开始，理解为什么二叉树需要 Rc<RefCell<TreeNode>>

问题：我想创建树后还能访问某个子节点

最简单的场景：

#![allow(unused)]
fn main() {
struct TreeNode {
    val: i32,
    left: Option<Box<TreeNode>>,
    right: Option<Box<TreeNode>>,
}

// 我想创建一棵树，并且保留对左子节点的访问
let left_child = Box::new(TreeNode { val: 2, left: None, right: None });
let root = TreeNode { 
    val: 1, 
    left: Some(left_child),  // ❌ left_child 的所有权移走了
    right: None 
};

// println!("{}", left_child.val);  // ❌ 编译错误：left_child 已经被 move
}

问题本质：我想在两个地方同时持有这个节点（root 持有，我也想持有），但所有权只能有一个。

这个需求自然吗？很自然：

  • 测试时想检查某个子节点
  • 算法中需要同时操作父节点和子节点
  • 想从树中“摘取“一个子树继续处理

解决方案 1：Rc<T> - 共享所有权

#![allow(unused)]
fn main() {
use std::rc::Rc;

struct TreeNode {
    val: i32,
    left: Option<Rc<TreeNode>>,
    right: Option<Rc<TreeNode>>,
}

// 现在可以共享了
let left_child = Rc::new(TreeNode { val: 2, left: None, right: None });
let root = TreeNode {
    val: 1,
    left: Some(Rc::clone(&left_child)),  // ✅ 克隆引用，不是克隆数据
    right: None,
};

println!("{}", left_child.val);  // ✅ 可以访问
println!("引用计数: {}", Rc::strong_count(&left_child));  // 输出: 2
}

Rc 做了什么？

  • 允许多个“所有者“
  • 通过引用计数追踪：最后一个所有者消失时才释放内存
  • Rc::clone() 只增加计数，不拷贝数据

新问题：如果我想修改节点呢？

#![allow(unused)]
fn main() {
use std::rc::Rc;

let left_child = Rc::new(TreeNode { val: 2, left: None, right: None });
let root = TreeNode {
    val: 1,
    left: Some(Rc::clone(&left_child)),
    right: None,
};

// 我想把左子节点的值改成 20
left_child.val = 20;  // ❌ 编译错误：Rc 提供的是不可变访问
}

为什么 Rc 不允许修改？

  • Rc 可能有多个引用同时存在
  • 如果允许修改，就相当于允许多个可变引用（违反借用规则）

但我确实需要修改，怎么办？

解决方案 2：Rc<RefCell<T>> - 共享 + 可修改

#![allow(unused)]
fn main() {
use std::rc::Rc;
use std::cell::RefCell;

struct TreeNode {
    val: i32,
    left: Option<Rc<RefCell<TreeNode>>>,
    right: Option<Rc<RefCell<TreeNode>>>,
}

// 创建节点
let left_child = Rc::new(RefCell::new(TreeNode { val: 2, left: None, right: None }));
let root = Rc::new(RefCell::new(TreeNode {
    val: 1,
    left: Some(Rc::clone(&left_child)),
    right: None,
}));

// ✅ 可以共享
println!("引用计数: {}", Rc::strong_count(&left_child));  // 2

// ✅ 可以修改
left_child.borrow_mut().val = 20;
println!("修改后的值: {}", left_child.borrow().val);  // 20

// 通过 root 也能看到修改
if let Some(left) = &root.borrow().left {
    println!("从 root 看到的值: {}", left.borrow().val);  // 20
}
}

RefCell 做了什么？

  • 把借用检查从编译时推迟到运行时
  • borrow() → 获取不可变引用
  • borrow_mut() → 获取可变引用
  • 如果违反规则（比如同时有可变和不可变引用）会在运行时 panic

为什么需要组合？一张图说清楚

能共享吗？能修改吗？
Box<TreeNode>❌✅
Rc<TreeNode>✅❌
RefCell<TreeNode>❌✅
Rc<RefCell<TreeNode>>✅✅

结论：如果你需要“多个地方持有同一个节点并且能修改它“，就必须用 Rc<RefCell<T>>。

实际例子：翻转二叉树

#![allow(unused)]
fn main() {
use std::rc::Rc;
use std::cell::RefCell;

type TreeLink = Option<Rc<RefCell<TreeNode>>>;

struct TreeNode {
    val: i32,
    left: TreeLink,
    right: TreeLink,
}

impl TreeNode {
    fn new(val: i32) -> Rc<RefCell<Self>> {
        Rc::new(RefCell::new(TreeNode {
            val,
            left: None,
            right: None,
        }))
    }
}

fn invert_tree(root: TreeLink) -> TreeLink {
    if let Some(node) = root {
        // 获取可变借用
        let mut node_mut = node.borrow_mut();
        
        // 交换左右子树
        let temp = node_mut.left.clone();
        node_mut.left = node_mut.right.clone();
        node_mut.right = temp;
        
        // 释放借用（重要！）
        drop(node_mut);
        
        // 递归翻转
        invert_tree(node.borrow().left.clone());
        invert_tree(node.borrow().right.clone());
        
        Some(node)
    } else {
        None
    }
}
}

关键点：

  1. 需要 borrow_mut() 才能修改节点
  2. 递归前要 drop(node_mut) 释放借用，否则递归时会 panic
  3. .clone() 克隆的是 Rc 引用，不是整棵树

常见操作速查

#![allow(unused)]
fn main() {
use std::rc::Rc;
use std::cell::RefCell;

let node = Rc::new(RefCell::new(TreeNode { val: 1, left: None, right: None }));

// 1. 读取值
let val = node.borrow().val;

// 2. 修改值
node.borrow_mut().val = 2;

// 3. 添加子节点
let left = TreeNode::new(10);
node.borrow_mut().left = Some(left);

// 4. 访问子节点
if let Some(left_node) = &node.borrow().left {
    println!("{}", left_node.borrow().val);
}

// 5. 克隆引用（不是深拷贝）
let another_ref = Rc::clone(&node);
println!("引用计数: {}", Rc::strong_count(&node));  // 2
}

注意事项

1. 运行时借用检查会 panic

#![allow(unused)]
fn main() {
let node = TreeNode::new(1);
let r1 = node.borrow();
let r2 = node.borrow_mut();  // ❌ panic: already borrowed
}

解决方法：及时释放借用

#![allow(unused)]
fn main() {
let node = TreeNode::new(1);
{
    let r1 = node.borrow();
    println!("{}", r1.val);
}  // r1 在这里被释放
let r2 = node.borrow_mut();  // ✅ 现在可以了
}

2. 循环引用会导致内存泄漏

#![allow(unused)]
fn main() {
// 父节点指向子节点，子节点又指向父节点
// 引用计数永远不会归零，内存泄漏！
}

解决方法：子节点指向父节点时用 Weak<T>

#![allow(unused)]
fn main() {
use std::rc::Weak;

struct TreeNode {
    val: i32,
    left: Option<Rc<RefCell<TreeNode>>>,
    right: Option<Rc<RefCell<TreeNode>>>,
    parent: Option<Weak<RefCell<TreeNode>>>,  // 用 Weak 打破循环
}
}

总结

Rc<RefCell<TreeNode>> 不是过度设计，而是表达了一个自然的需求：

我需要在多个地方持有同一个树节点，并且能修改它。

  • Rc = “这个节点的生命由引用计数管理，可以有多个所有者”
  • RefCell = “我需要在运行时修改它，我保证不违反借用规则”

如果你的算法只是遍历树，用 &TreeNode 借用就够了。
如果需要修改树结构或者持有子树，那 Rc<RefCell<T>> 是最自然的选择。

多线程并发

本节主要汇总一些rust中多线程相关的知识点

线程管理

std::thread::spawn(func)

spawn用于开辟一个新的子线程，这个子线程用来执行其内部的闭包函数func，其中func: FnOnce() -> ……，这意味着func不应带参数
同时，它会返回一个JoinHandle<T>类型的变量，其中 T 是func返回值的类型，在子线程运行结束后才可以被获取到

JoinHandle<T>::join()

join()用来等待JoinHandle<T>的值被拿到，并输出Ok(T)
但是如果子线程执行过程中发生panic!，显然就等不到JoinHandle<T>了，此时join()会返回一个Err(e)，e是子线程崩溃的错误信息

move

如果spawn(func)里的func是普通的闭包，那么其会遵循最小特权原则对捕获变量进行借用，但是子线程的存活时间是不一定的，因此spawn要求func的借用必须是’static，否则就会报错；
为了避免这种情况发生，需要在func前加上move，他会将捕获的变量以和函数传入参数同样的方式进行copy或move，例如对于i32，他会Copy一份新的i32数据供子线程使用，而对于String，他会直接把所有权转移到子线程内部，主线程将不能再使用。

共享状态

std::sync::Arc<T>

到目前为止，子线程还无法和主线程或其它子线程共享数据，因为数据要么被Copy要么被Move。
为了能够共享一份数据，需用Arc::new()将T类型的数据包裹起来得到Arc<T>，它相当于为数据开了后门，让T可以在多个地方同时拥有所有权，同时给数据旁边添加了一个计数器，每多一个变量拥有它的所有权计数器就加一，它的安全性由Arc的原子性保证； 对于得到的Arc<T>类型变量，可以直接当作T类型的变量使用，而使用Arc::clone(&var: Arc<T>)能够克隆一份同样是Arc<T>类型的、拥有数据所有权的变量，同时该数据的计数器加一，下面是一个例子：

use std::thread;
use std::sync::Arc;

fn main() {
    let s = Arc::new(String::from("123"));  // 创建Arc<String>,同时为数据创建计数器
    
    for _ in 0..5 {
        let s_clone = Arc::clone(&s);    // Arc<String>的所有权被共享给新变量，计数器加一；同时它等价于s.clone();
        thread::spawn(move || {
            println!("{s_clone:?}");    //s_clone: Arc<String>的使用与String相同
        });
    }
}

尽管如此，Arc<T>只能实现共享读，对于共享写，需要使用锁

std::sync::Mutex<T>、Mutex<T>::lock()、MutexGuard<'a, T>

Mutex<T>相当于把T类型的数据装进Mutex这个盒子里，一个线程只有成功打开了Mutex，该线程才能对数据进行改写； Mutex<T>::lock()就是那个开锁/上锁的工具，它会阻塞等待，显然开锁成功后数据又会被立即上锁； Mutex<T>::lock()后的返回值是Result<MutexGuard<'a, T>, PoisonError>，其中MutexGuard<'a, T>能够实现对数据进行监控，从而实现自动释放锁，使得锁可以被其它线程打开。

use std::thread;
use std::sync::{Arc, Mutex};

fn main() {
    let s = Arc::new(Mutex::new(String::from("Hello!")));    // 经典的Arc<Mutex<T>>结构
    let mut handles = Vec::new();
    for _ in 0..5 {
        let s_clone = Arc::clone(s);
        let handle = thread::spawn(move || {
            let s_guard = s_clone.lock().unwrap();  // 阻塞等待拿锁
            let s_rev = (*s_guard).chars().rev().collect(); // 这里s_guard解引用得到原字符串
            // let s_rev = s_guard.chars().rev().collect(); 自动解引用也可以达到同样的效果
            println!("{s_rev:?}");
        }); //自动释放锁
        
        handles.push(handle);
    }
    
    for handle in handles {
        handle.join().unwrap();
    }
}

std::sync::RwLock<T>、RwLock<T>::{read(), write()}

与Mutex一样是为了实现共享数据的读写，但是它把lock拆成了read和write两部分，对读写有着不同的处理：

有人在读 (Read Lock)：

其他想读的人：直接放行。于是内存里可以同时存在无数个“只读中间人”。
其他想写的人：原地等待。必须等所有读的人都走了，才能进去。

有人在写 (Write Lock)：

无论是想读还是想写的人：全部等待。
简而言之，一个线程的写与其它线程的读写都互斥

至此，我们实现了对数据的共享读写

消息传递

std::sync::mpsc::{channel(), Receiver, Sender }, Receiver::recv(), Sender::send()

异步

I/O 密集型的任务如果直接交给一个线程执行，那么这个线程会阻塞且占用系统资源，使得 CPU 无法高效利用。Rust 提供的异步编程模型在需要等待时会直接挂起当前任务，并将控制权交给异步运行时（最常用的是 Tokio），这样 CPU 就可以继续执行其他任务。

Rust 异步模型提供了 async和 await 两个语法糖来实现异步模型，async 用于定义一个异步函数或异步块， await 用于等待一个异步操作完成并获取结果。

.await 只能在 async 函数或 async 块中使用，不能在普通函数中使用（async main需要使用#[tokio::main]）。如需在普通函数中等待异步操作，可以使用 block_on 函数来运行一个 Future 并等待其完成。

使用这两个语法糖时，不需要手写实现 Future trait，事实上写下 async 就是让编译器自动为你生成实现 Future trait 的代码。Future 的核心是一个必须方法 poll：

#![allow(unused)]
fn main() {
enum Poll<T> {
    Ready(T),
    Pending,
}

trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}
}

原理

async 根据异步函数中的每个 await 进行切割生成一个枚举状态机，每个 await 都会将当前状态（之后还会用到的变量）保存到一个状态变量中，并在下次调用 poll 时恢复状态继续执行。

而 poll 是状态机的推进器，每次调用 poll 都会检查当前状态并执行相应的代码。至于 poll 的参数 cx 是一个上下文对象，包含了当前任务的 waker（唤醒器），当异步操作完成时，可以通过 waker 来通知运行时继续执行该任务。

waker 可以理解为“唤醒任务的句柄”。在 poll 返回 Pending 后，运行时会把它和底层 I/O 事件关联起来；当事件就绪时，再通过 waker 把任务放回就绪队列。

executor（实际还有reactor） 是一个调度器/执行器，它检查就绪队列内有没有任务，有的话就调用 poll 来推进状态机。

总结一下

waker, executor, poll 是 Rust 异步编程的核心，分别负责唤醒任务、调度任务、更新状态机。async 语法本身由编译器转换为状态机；手写实现 Future 时，本质上是在自己维护这套状态推进逻辑。
通过 async 和 await 语法糖，Rust 提供了一个高效且易用的异步编程模型，使得开发者能够轻松编写高性能的异步代码。

进阶：为什么会看到 Pin

看到 poll(self: Pin<&mut Self>, ...) 时，很多人会疑惑：为啥不是普通的 &mut self？

一句话理解：Pin 约束的是“这个值一旦被固定，就不能再随便移动它在内存里的位置”。

这在 async 里很关键，因为 Future 会被多次 poll，每次都要接着上一次状态继续跑。为了保证状态里那些跨 await 保存的数据关系不被破坏，poll 接口就要求在一个被 pin 住的可变引用上推进。

具体是这样的:

#![allow(unused)]
fn main() {
pub struct Pin<P> {
    pointer: P, // pointer 不是 pub 的
}
}

Pin 是一个实现了 Deref 和 DerefMut 的智能指针，他内部包裹了另外一个指针P，并且只要P指针指向的内容（我们称为T）没有实现Unpin，则可以保证T永远不会被移动

I/O 多路复用

I/O 多路技术是指在单线程中同时监听多个 I/O 事件，当其中一个事件就绪时，程序就可以处理它，而不需要为每个 I/O 操作创建一个线程。Rust 的异步编程模型就是基于这种技术实现的。

在 Linux 上，最广泛使用的 I/O 多路复用技术是 epoll，它将需要监听的文件描述符注册到内核里的事件表中，这个表底层实现是一个红黑树。当事件发生时，就绪的文件描述符会被放入一个就绪队列中，程序可以调用 epoll_wait 来等待事件的发生并获取就绪的文件描述符。

比起 poll，epoll 的优势在于它的事件表是内核维护的，程序只需要注册一次就可以持续监听，而不需要手动维护一张 fd 表并在每次都传入整个文件描述符集合，这样就大大减少了系统调用的开销，提高了性能。

为什么提到 I/O 多路复用

Rust 的异步编程模型是基于 I/O 多路复用技术实现的，异步运行时（如 Tokio）内部使用 epoll 来监听多个 I/O 事件。当一个异步操作（如网络请求或文件读写）需要等待时，运行时会将其注册到 epoll 中，并在事件就绪时通过 waker 唤醒对应的任务继续执行。

宏

本节主要介绍rust中的宏，它实现了代码->代码的转换，可以用来简化代码，提高代码的可读性。

宏的分类

rust中的宏分为两种，一种是声明式宏，另一种是过程宏。

过程宏

过程宏分为三种类型：

1. 派生宏 (Derive Macros)

用于自动实现trait。

#![allow(unused)]
fn main() {
// 在 lib.rs 中
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, DeriveInput};

#[proc_macro_derive(HelloMacro)]
pub fn hello_macro_derive(input: TokenStream) -> TokenStream {
    let input = parse_macro_input!(input as DeriveInput);
    let name = &input.ident;
    let expanded = quote! {
        impl HelloMacro for #name {
            fn hello_macro() {
                println!("Hello, Macro! My name is {}!", stringify!(#name));
            }
        }
    };
    TokenStream::from(expanded)
}
}

}

宏的卫生性

Rust宏是“卫生“的，意味着宏中定义的变量不会与调用处的变量冲突：

macro_rules! using_a {

        {
            let a = 42;

        }
    };
}

fn main() {
    let a = 1;
    let result = using_a!(a + 2);  // 这里a = 1，而不是42
    println!("Result: {}", result);  // 输出: 3
}

迭代器及相关内容

作为函数式风格语言，rust提供了非常强大的迭代器功能，也是rust中极其优雅的处理方式之一；迭代器往往作用于Vec、HashMap等集合类型上，能够让我们以一种非常简洁的方式对集合进行各种操作，如转换、过滤、累积等。

迭代器的两个重要 trait ：Iterator 和 IntoIterator

若需要为某个类型实现迭代器，需要实现 Iterator trait ，其必要方法是 next() ，用于获取迭代器中的下一个元素，如果迭代器中还有元素，则返回Some(元素)，否则返回None。

#![allow(unused)]
fn main() {
trait Iterator {
    type Item;
    fn next(&mut self) -> Option<Self::Item>;
}
}

如需将某个集合转换为迭代器，则需要实现 IntoIterator trait，必要方法为 into_iter() ，它返回一个生成的迭代器。

#![allow(unused)]
fn main() {
trait IntoIterator {
    type Item;
    type IntoIter: Iterator<Item = Self::Item>;
    fn into_iter(self) -> Self::IntoIter;
}
}

从集合生成迭代器

迭代器只用于迭代，而数据一般存储在集合中，所以首先需要从集合生成迭代器（对应于 IntoIterator trait），下面是一些常见的从集合生成迭代器的方法：

Vec -> 迭代器

iter , iter_mut 和 into_iter

iter 方法返回对集合的不可变引用的迭代器

iter_mut 方法返回对集合的可变引用的迭代器，使用iter_mut后，可以修改集合中的元素，但原来的集合仍然可用

into_iter 方法则返回集合的所有权的迭代器，使用into_iter后，原来的集合将不再可用

方法Item类型用途
iter&T只读遍历
iter_mut&mut T原地修改集合元素
into_iterT消费集合，拿到所有权

字符串 -> 迭代器

字符串 String 是一个特殊的存在，它的迭代方式很不同而且非常丰富

chars 和 bytes: 从字符串生成的字符迭代器

字符串的迭代与集合有所不同，它不拿到字符串的引用作为元素，而是直接迭代字符串中的字符或字节

chars 按照 utf-8 编码返回每个字符的迭代器

bytes 直接按字节返回迭代器。有趣的是, as_bytes 返回字符串的字节切片，想要迭代器的话可以使用as_bytes().iter()

rust中的字符串是UTF-8编码的，所以一个字符可能由多个字节组成, chars 方法会正确处理这些字符，而 bytes 方法只返回每个字节的迭代器。

lines 和 split: 从字符串生成元素为字符串迭代器

lines 将字符串按行分割，返回一个迭代器，每个元素是字符串的一行

split 根据指定分隔字符串将字符串分割，返回一个迭代器，每个元素是分割后的字符串片段，此外还有 split_whitespace, split_inclusive 等变体

matches: 从字符串生成匹配模式迭代器

matches 方法传入一个pattern（字符串、正则表达式等），返回一个迭代器，包含字符串中所有匹配指定模式(如果只找第一个，可以使用find或position方法，尽管他俩传入的参数应该是一个返回值为bool的闭包)的子串。

chars_indices , bytes_indices 和 match_indices: 从字符串生成带索引的迭代器

这几个相当于上面几个方法的带索引版本，返回的迭代器中的元素是一个元组，包含元素的索引和元素本身。

HashMap/HashSet -> 迭代器

iter 和 into_iter: 从HashMap生成键值对的迭代器

它们也可以用于HashMap和HashSet，和Vec的实际上一样，区别只在于HashMap返回键值对元组

keys 和 values: 从HashMap生成键和值的迭代器

keys 返回一个包含HashMap中的所有键的引用的迭代器
values 返回一个包含HashMap中的所有值的引用的迭代器

迭代器上的常用方法

迭代器上的方法往往有两类，分别是适配器和消费器，适配器构造新迭代器，消费器则调用 next 推动迭代器运行（对应于 Iterator trait），得到一个最终结果。

迭代器 -> 迭代器（迭代器适配器）

迭代器适配器通常是惰性的，只是描述了一串转换过程，只有遇到消费器时才会真正执行
以下这些方法可以通过一个迭代器返回新的迭代器，用来实现各种转换

map ， filter 和 filter_map

map 用于对迭代器中的每个元素进行转换，返回一个新的迭代器

filter 则用于过滤迭代器中的元素，返回一个新的迭代器，只包含满足条件的元素

filter_map 则是map和filter的结合体，能够过滤的同时进行转换

注意：map和filter方法返回的都是新的迭代器，原来的迭代器并没有被修改

rev: 翻转操作，无需多言

zip 和 unzip

zip 用于将两个迭代器进行配对，返回一个新的迭代器，包含两个迭代器中对应位置的元素组成的元组，如：

#![allow(unused)]
fn main() {
let zipped = iterator1.iter().zip(iterator2.iter());
}

unzip 作用于元素为二元组的迭代器，拆分成两个迭代器，并包裹成一个元组返回。

enumerate

enumerate 用于将迭代器中的元素进行枚举，返回一个新的迭代器，包含元素的索引和元素本身组成的元组，如：

#![allow(unused)]
fn main() {
for (index, value) in iterator.iter().enumerate() {
    println!("Index: {}, Value: {}", index, value);
}
}

chain

chain 用于将两个迭代器连接起来，返回一个新的迭代器，包含两个迭代器中的所有元素，如：

#![allow(unused)]
fn main() {
let chained = iterator1.iter().chain(iterator2.iter());
}

scan

scan 用于对迭代器中的元素进行累积操作，返回一个新的迭代器，包含每次累积的结果，如：

#![allow(unused)]
fn main() {
let scan_result:Vec<_> = iterator.iter().scan(0, |acc, &x| {
    *acc += x;
    Some(*acc)
})
.collect();
}

scan接受的两个参数中，第一个是积累值的初始值，第二个是一个返回值为Option类型的闭包，闭包又接受两个参数，第一个是积累值的可变引用，第二个是迭代器中的元素；
迭代到某一步时，若返回Some(值)，则将该值作为新的迭代器中的元素；若返回None，则迭代直接结束。

take 和 skip

take 取前n个元素作为一个新的迭代器返回 skip 则跳过前n个元素，取剩下的元素作为迭代器返回

step_by

step_by 指定迭代器的步长，从而得到一个跳过一些元素的新迭代器

cloned 和 copied

他俩主要是为了把原来迭代器引用的元素拿出来转成值的元素，形成一个新迭代器

迭代器 -> 其它 （迭代器消费器）

消费器会真正推动迭代器运行，可能消费部分元素，也可能消费完整个迭代器，它们常常通过能够通过迭代器得到一个最终结果

collect

collect 是rust中最强大的迭代器方法之一，用于将迭代器中的元素收集到一个其它存储结构中。它可以将迭代器中的元素收集到一个Vec、HashMap、HashSet等集合类型中，也可以收集到一个字符串中，甚至可以收集到一个自定义的结构体中。使用collect方法时，需要指定要收集到的目标类型。

nth

nth 用于获取迭代器中第n个元素，返回一个Option类型的值，如果迭代器中有第n个元素，则返回Some(元素)，否则返回None。

find 和 position

find 在迭代器中查找满足条件的第一个元素，如果找到满足条件的元素，则返回Some(元素)，否则返回None。 position 用迭代器中查找满足条件的第一个元素的位置，如果找到满足条件的元素，则返回Some(位置)，否则返回None。

any 和 all

显然，这两个方法是来判断迭代器中的元素是否存在或全都满足某个条件的，返回一个bool值。

fold 和 reduce

fold 用于将迭代器中的元素进行累积，返回 一个最终的结果

reduce 同样也是累积，与fold方法不同的是，reduce方法没有初始值，而是将迭代器中的第一个元素作为初始值,并且它的返回值是Option类型，None用来应对空迭代器的情况。

#![allow(unused)]
fn main() {
// 将vec中的元素进行累积，最终得到sum的值为6。
// fold 实现
let sum1 = vec![1, 2, 3].into_iter().fold(0, |acc, x| acc + x); // sum1 == 6
// reduce 实现
let sum2 = vec![1, 2, 3].into_iter().reduce(|acc, x| acc + x); // sum2 == Some(6)
}

for 循环

for 循环也是一种消费迭代器的方式，能够对迭代器中的每个元素执行某些操作，如打印、修改等。

#![allow(unused)]
fn main() {
for element in &mut iterator {
    element.do_something();
}
}

除了上面提到的方法外，还有很多其他的迭代器方法，如 max 、 min 、 sum 、 count 、 eq 、cmp、is_sorted、product(累乘) 等等，通过名称可以轻易推断它们的功能，也就不多赘述了。

rust 异步 中介绍了 Rust 的异步编程模型和 Future 的概念，这里主要是对 Tokio 的介绍

tokio

1. 运行时 Runtime

tokio 提供异步运行时，负责调度任务、I/O 事件和定时器。

#[tokio::main]
async fn main() {
	// async 入口
}
  • 常用运行时模式:
    • multi_thread: 默认，多线程调度。
    • current_thread: 单线程调度，适合轻量场景。

2. 任务 Task

#![allow(unused)]
fn main() {
let handle = tokio::spawn(async {
	1 + 2
});

let v = handle.await?;
}
  • tokio::spawn 要求任务是 'static + Send（多线程运行时）。
  • 任务取消: handle.abort()。

3. 并发组合

#![allow(unused)]
fn main() {
let (a, b) = tokio::join!(task_a(), task_b());
let (x, y) = tokio::try_join!(op_x(), op_y())?;
}
  • join!: 等全部完成，保留各自结果。
  • try_join!: 任一返回错误则整体返回错误。

4. 超时与 select

#![allow(unused)]
fn main() {
use tokio::time::{timeout, Duration};

let r = timeout(Duration::from_secs(2), slow_op()).await;
}
#![allow(unused)]
fn main() {
tokio::select! {
	_ = shutdown.recv() => {},
	msg = rx.recv() => {
		// 处理消息
	}
}
}
  • timeout 用于给异步操作设置时限。
  • select! 用于等待多个异步分支，谁先完成就走谁。

5. 异步通道

#![allow(unused)]
fn main() {
use tokio::sync::{mpsc, oneshot};

let (tx, mut rx) = mpsc::channel::<String>(64);
let (done_tx, done_rx) = oneshot::channel::<u32>();
}
  • mpsc: 多生产者单消费者。
  • oneshot: 一次性响应，常用于请求-响应。

6. 同步原语

  • tokio::sync::Mutex / RwLock: 异步锁。
  • Semaphore: 并发限流。
  • Notify: 轻量通知。

注意: 尽量避免在持锁期间做耗时 await。

7. 阻塞代码

#![allow(unused)]
fn main() {
let v = tokio::task::spawn_blocking(|| {
	heavy_cpu_or_blocking_io()
}).await?;
}
  • CPU 密集或阻塞操作放入 spawn_blocking。
  • 不要在 async 任务里直接长时间阻塞线程。

8. 常见坑

  • 忘记 .await，future 实际没有执行。
  • 在 async 任务中使用 std::sync::Mutex 并跨 await 持锁。
  • 无限制 spawn 导致内存和调度压力过大。
  • 任务里捕获了非 'static 引用，导致无法 spawn。

9. 学习顺序建议

  1. spawn + join! + try_join!
  2. mpsc + oneshot
  3. select! + timeout
  4. Semaphore + spawn_blocking

VS Code 终端一打开就自动换行刷屏：一次完整排查与修复

问题现象

我在 Windows 上使用 VS Code 时，终端一打开就出现“自动换行 + 刷屏”，几乎无法正常输入命令。

典型表现：

  • 新开终端后不断自动换行刷屏

关键报错（开发者工具）

开发者工具里出现了这类错误：

Uncaught TypeError: Cannot read properties of undefined (reading 'dimensions')
... at get dimensions (xterm.js)
... _refreshCanvasDimensions
... _refreshDecorations

以及 VS Code 日志中的同类报错：

ERR Cannot read properties of undefined (reading 'dimensions')
... node_modules/@xterm/xterm/lib/xterm.js

根因判断

这类错误基本可归因于：

  • VS Code 内置终端（xterm.js）在“终端尺寸计算/重绘”链路发生异常
  • dimensions 对象在某些时机是 undefined，触发连续重绘
  • 在部分环境下，GPU 渲染、Shell Integration、终端持久会话等功能叠加后更容易触发

重点：这通常不是你的 PowerShell 命令本身死循环，而是终端渲染层异常。

我的修复方案（稳定优先）

我采用了“先止血，再优化”的思路：先让终端恢复可用。

在 VS Code 用户设置 settings.json 中加入/修改：

{
  "terminal.integrated.defaultProfile.windows": "PowerShell",
  "terminal.integrated.gpuAcceleration": "off",
  "terminal.integrated.shellIntegration.enabled": false,
  "terminal.integrated.smoothScrolling": false,
  "terminal.integrated.enablePersistentSessions": false
}

为什么这组配置有效

  • 关闭 GPU 渲染：绕开部分显卡/驱动/渲染路径导致的尺寸计算异常
  • 关闭 Shell Integration：减少终端注入与渲染状态耦合
  • 关闭平滑滚动：避免频繁动画重绘
  • 关闭持久会话：避免恢复历史终端状态时触发异常状态
  • 固定默认终端为 PowerShell：减少 profile/终端类型切换变量

验证步骤

  1. 执行 Developer: Reload Window
  2. 关闭所有终端标签后重新打开
  3. 连续开关终端 3~5 次，观察是否仍刷屏

如果 10~30 秒内不再出现自动换行刷屏，说明修复生效。

进一步优化（按需逐项恢复）

如果你想要更高性能或更多功能，可以按顺序“逐项恢复”，每次只改一项并观察：

  1. 先恢复 terminal.integrated.shellIntegration.enabled 为 true
  2. 再恢复 terminal.integrated.enablePersistentSessions 为 true
  3. 最后尝试 terminal.integrated.gpuAcceleration（若复发，立即改回 off）

这样可以精确定位触发项，避免一次性改动太多。

经验总结

  • 看到 xterm.js + dimensions 报错时，优先怀疑终端渲染链路，而不是 shell 脚本
  • 终端问题排查优先级建议：
    1. 先稳定可用（降级配置）
    2. 再逐项恢复（找根触发点）
    3. 最后考虑扩展冲突与 VS Code 版本升级

如果你也遇到“终端一开就刷屏”，可以直接先用上面的稳定配置，通常能很快恢复工作流。
