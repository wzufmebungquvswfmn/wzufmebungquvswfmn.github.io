# 操作系统与 Rust 系统编程

系统编程的主线是：

> 通过操作系统提供的抽象和接口，管理 CPU、内存、文件、设备、网络和并发。

这篇笔记分成两部分：

1. **操作系统基础**：先把内核、系统调用、进程、线程、虚拟内存、文件描述符、IPC、同步、信号、I/O 模型这些概念讲清楚。
2. [**Rust 系统编程**](#第二部分rust-系统编程)：再看 Rust 如何使用这些操作系统能力，包括 `std` 封装、Unix fd、`libc`、FFI、`mmap`、`unsafe`、`no_std`、异步运行时和诊断工具。

这样安排的目的很简单：先建立 OS 的地基，再解释 Rust 为什么这样设计、哪些 API 其实是在包一层系统调用。

## 用户态与内核态

现代操作系统把程序运行分成不同的特权级。普通应用运行在**用户态**，内核运行在**内核态**。

- **用户态**：不能执行特权指令，不能随意访问内核内存，不能直接操作硬件。
- **内核态**：可以管理页表、中断、调度器、设备驱动和所有进程资源。

这个分层是操作系统安全性的基础。否则一个普通程序就可能改写别的进程内存、关闭中断、破坏文件系统，甚至让整个机器崩溃。

用户程序需要内核服务时，不能直接调用内核函数，而是通过**系统调用**进入内核。内核完成工作后再返回用户态。

```text
用户程序
  |
  |  系统调用
  v
内核：检查权限、访问资源、返回结果
  |
  v
用户程序继续执行
```

**Rust 对应：** `std::fs::File::open`、`TcpStream::connect`、`thread::spawn` 等高层 API 最终都会走到操作系统能力上。Rust 通常不让你直接碰内核，而是通过标准库或 `libc` 间接使用系统调用。

## 系统调用

系统调用是用户程序请求内核服务的正式入口。

常见系统调用类别：

| 类别 | Linux 示例 |
| --- | --- |
| 进程控制 | `fork`、`execve`、`exit`、`waitpid` |
| 文件 I/O | `open`、`read`、`write`、`close`、`fsync` |
| 内存管理 | `mmap`、`munmap`、`brk`、`mprotect` |
| 网络 | `socket`、`bind`、`listen`、`accept`、`connect` |
| IPC | `pipe`、`shmget`、`semop`、`sendmsg` |
| 信号 | `kill`、`sigaction`、`sigprocmask` |

一次系统调用大致经历这些步骤：

1. 用户程序按系统调用 ABI 放好系统调用号和参数。
2. CPU 执行 `syscall`、`int 0x80`、`svc`、`ecall` 等陷入指令。
3. CPU 从用户态切到内核态，进入内核的系统调用入口。
4. 内核检查参数、权限和资源状态。
5. 内核执行对应操作，并把结果或错误码返回给用户程序。

系统调用不是普通函数调用。它会跨越用户态和内核态边界，通常比普通函数调用更贵。因此高性能程序会减少不必要的系统调用，例如批量读写、使用缓冲、使用 `epoll` 避免大量阻塞线程。

### ABI 与错误

系统调用 ABI 规定了调用号、参数和返回值放在哪里。以 Linux x86_64 为例：

| 项目 | 位置 |
| --- | --- |
| 系统调用号 | `rax` |
| 参数 | `rdi, rsi, rdx, r10, r8, r9` |
| 返回值 | `rax` |
| 触发指令 | `syscall` |

系统调用失败时通常返回负错误码，用户态库会把它转换成更友好的错误形式，例如 C 的 `errno`，Rust 的 `std::io::Error`。

**Rust 对应：** 优先使用标准库。只有在标准库没有暴露某个 OS 能力时，才考虑 `libc`、`nix`、`rustix` 或手写 FFI。

## 进程

进程是操作系统分配资源的基本单位。一个进程通常拥有：

- 独立的虚拟地址空间。
- 文件描述符表。
- 当前工作目录、环境变量、权限信息。
- 进程 ID、父进程 ID、进程状态。
- 一个或多个线程。

进程之间默认不共享内存。这个隔离非常重要：一个进程崩溃，通常不会直接破坏另一个进程的地址空间。

```text
进程 A                           进程 B
+-------------------+           +-------------------+
| 地址空间           |           | 地址空间           |
| fd 表              |           | fd 表              |
| 权限 / 环境变量     |           | 权限 / 环境变量     |
+-------------------+           +-------------------+
        |                               |
        +----------- IPC ---------------+
```

### 创建与替换

Unix 里经典的进程创建模型是：

- `fork`：复制当前进程，产生子进程。
- `execve`：把当前进程映像替换成另一个可执行文件。
- `waitpid`：父进程等待子进程结束并回收状态。

Shell 执行命令时大致就是 `fork` 一个子进程，然后在子进程中 `execve` 目标程序。

**Rust 对应：** Rust 标准库通常不直接暴露 `fork` 风格的接口，而是用 `std::process::Command` 表达“启动一个新程序”。这比裸 `fork` 更符合跨平台抽象。

## 线程与调度

线程是 CPU 调度的基本执行单元。同一个进程内的多个线程：

- 共享地址空间。
- 共享文件描述符表。
- 各自拥有独立的栈和寄存器上下文。
- 一个线程崩溃可能导致整个进程崩溃。

进程和线程的区别：

| 特性 | 进程 | 线程 |
| --- | --- | --- |
| 地址空间 | 独立 | 同进程内共享 |
| 创建成本 | 较高 | 较低 |
| 切换成本 | 较高 | 较低 |
| 通信方式 | IPC | 共享内存加同步 |
| 隔离性 | 强 | 弱 |

调度器负责决定下一个运行哪个线程。线程可能处于运行、就绪、阻塞、结束等状态。

```text
就绪队列 -> 运行 -> 阻塞等待 I/O
   ^          |          |
   |          v          |
   +------ 时间片结束 <- I/O 完成
```

### 上下文切换

上下文切换就是保存当前执行流的 CPU 状态，并恢复另一个执行流的状态。需要保存的内容通常包括：

- 程序计数器。
- 栈指针。
- 通用寄存器。
- 地址空间相关状态。
- 调度器维护的线程状态。

上下文切换有成本，所以高并发程序不一定是线程越多越好。线程太多会带来调度开销、栈内存开销和缓存失效。

**Rust 对应：** `std::thread::spawn` 创建的是 OS 线程。Rust 的 `Send` 和 `Sync` 不是 OS 概念，而是 Rust 在类型系统层面对跨线程移动和共享做出的安全约束，适合放到第二部分讲。

----

## 协程与用户态任务

协程不是操作系统必须提供的基本调度单位。它通常由语言运行时或库在用户态调度。

协程的特点：

- 数量可以远多于 OS 线程。
- 切换通常比 OS 线程便宜。
- 适合大量 I/O 等待任务。
- 如果协程长期不让出执行权，其他协程可能无法运行。

协程和线程的核心区别：

| 项目 | OS 线程 | 协程 / 用户态任务 |
| --- | --- | --- |
| 调度者 | 内核 | 运行时 / 程序 |
| 抢占 | 通常可抢占 | 常见为协作式 |
| 创建成本 | 较高 | 较低 |
| 适合场景 | CPU 并行、阻塞调用 | 大量 I/O 并发 |

**Rust 对应：** Rust 的 `async/await` 生成的是可轮询的 `Future`，需要 Tokio、async-std 等运行时驱动。它不是自动创建 OS 线程。

## 虚拟内存

虚拟内存让每个进程都以为自己拥有一整块连续、独立的地址空间。内核和硬件共同把虚拟地址翻译成物理地址。

虚拟内存带来的好处：

- **隔离**：进程不能直接访问其他进程内存。
- **简化**：程序使用连续地址，不需要关心物理内存碎片。
- **保护**：代码段可以只读，栈可以不可执行。
- **按需加载**：真正访问页面时才把数据加载进物理内存。
- **共享**：多个进程可以把同一个文件映射到各自地址空间。

典型进程地址空间：

```text
低地址
+------------------+
| 代码段 .text      |
| 只读数据 .rodata  |
| 数据段 .data/.bss |
| 堆 heap           |
|        ...        |
| 栈 stack          |
| 环境变量 / 参数    |
+------------------+
高地址
```

### 堆与栈

- **栈**：用于函数调用、局部变量、返回地址。分配和释放很快，但空间有限。
- **堆**：用于动态分配对象。生命周期更灵活，但需要分配器管理。

栈溢出、堆碎片、悬垂指针、越界访问等问题都和内存管理密切相关。

**Rust 对应：** Rust 的所有权、借用和生命周期主要是在语言层减少悬垂指针、重复释放、数据竞争等问题。但虚拟内存、页表、缺页这些仍然由操作系统管理。

----

## 分页、页表、TLB

分页是虚拟内存的常见实现方式。虚拟地址空间被切成固定大小的**页**，物理内存被切成同样大小的**页框**。

虚拟地址通常拆成：

```text
虚拟地址 = 页号 + 页内偏移
```

页号通过页表翻译成物理页框号，页内偏移保持不变。

```text
虚拟地址
  |
  v
页表：虚拟页号 -> 物理页框号
  |
  v
物理地址
```

### 缺页中断

当进程访问的虚拟页还没有映射到物理内存时，会触发缺页中断。内核可能会：

- 从磁盘加载对应页面。
- 分配新的物理页。
- 建立页表映射。
- 如果访问非法，则向进程发送异常，例如段错误。

### TLB

页表在内存里，直接查页表很慢。CPU 使用 TLB 缓存最近的地址翻译结果。

- TLB 命中：快速得到物理地址。
- TLB 未命中：需要查页表。
- 修改页表后：需要让相关 TLB 项失效。

**Rust 对应：** 普通 Rust 程序不能直接改页表，但可以通过 `mmap`、`mprotect`、分配器等接口影响虚拟内存布局。

## 文件描述符

Unix/Linux 把很多 I/O 资源统一抽象成文件描述符。fd 是进程 fd 表中的一个整数索引。

常见 fd：

| fd | 含义 |
| --- | --- |
| 0 | 标准输入 |
| 1 | 标准输出 |
| 2 | 标准错误 |

文件、管道、socket、终端、设备都可以通过 fd 读写。

```text
进程 fd 表
+----+----------------+
| 0  | stdin          |
| 1  | stdout         |
| 2  | stderr         |
| 3  | file.txt       |
| 4  | socket         |
+----+----------------+
```

### fd 与内核对象

fd 只是进程内的句柄。多个 fd 可以指向同一个内核文件对象，例如 `dup` 复制 fd 后就是这样。进程退出时，内核会关闭它持有的 fd。

需要注意：

- fd 编号会复用，关闭 `3` 后下次打开文件可能又得到 `3`。
- fd 泄漏会耗尽进程或系统资源。
- `fork` 后子进程会继承父进程的 fd，除非设置 close-on-exec。

**Rust 对应：** `std::fs::File` 在 Unix 上内部持有 fd，并在 `Drop` 时关闭。`OwnedFd`、`BorrowedFd`、`RawFd` 用来表达 fd 的所有权和借用关系。

## 进程间通信

进程之间默认内存隔离，所以需要 IPC 机制交换数据。

| 机制 | 特点 | 适合场景 |
| --- | --- | --- |
| 管道 | 字节流、单向、简单 | 父子进程、命令管线 |
| 命名管道 | 有路径名的管道 | 无亲缘关系进程 |
| Unix domain socket | 支持双向通信和传 fd | 本机服务通信 |
| TCP/UDP socket | 跨机器通信 | 网络服务 |
| 共享内存 | 避免拷贝，速度快 | 大数据交换 |
| 信号量 | 计数同步原语 | 进程间同步 |
| 消息队列 | 消息边界明确 | 任务队列、事件传递 |
| 信号 | 异步通知 | 中断、终止、 reload |

不同 IPC 的核心取舍：

- 管道和 socket 容易用，但数据要经过内核缓冲。
- 共享内存快，但必须自己处理同步。
- 信号轻量，但能做的事情非常有限。

**Rust 对应：** 标准库提供了进程管道、TCP/UDP、Unix socket 等封装。共享内存、信号量、消息队列通常需要第三方 crate 或 `libc`。

## 同步机制

只要多个执行流访问共享状态，就可能需要同步。

典型问题：

- **数据竞争**：多个线程同时读写同一块内存，且至少一个是写。
- **竞态条件**：结果依赖不可控的执行顺序。
- **死锁**：多个执行流互相等待对方释放资源。
- **饥饿**：某些执行流长期得不到资源。

常见同步原语：

| 原语 | 作用 |
| --- | --- |
| Mutex | 一次只允许一个执行流进入临界区 |
| RwLock | 多读单写 |
| Semaphore | 控制资源计数 |
| Condition Variable | 等待某个条件成立 |
| Atomic | 无锁地读改写小数据 |
| Barrier | 等待一组线程都到达某点 |

### 死锁条件

经典死锁通常同时满足四个条件：

1. 互斥：资源不能被同时占用。
2. 持有并等待：持有一个资源时继续等待其他资源。
3. 不可抢占：资源不能被强制夺走。
4. 循环等待：多个任务形成等待环。

打破任意一个条件都可以避免死锁。工程上常见做法是固定加锁顺序、缩小锁作用域、避免持锁执行阻塞 I/O。

**Rust 对应：** Rust 类型系统能防止很多内存层面的数据竞争，但不能自动消除所有逻辑竞态和死锁。`Mutex`、`RwLock`、`Atomic` 的具体使用放到第二部分讲。

## 信号

信号是 Unix 提供的异步事件通知机制。它可以打断进程当前执行流，让进程终止、暂停、继续，或者执行信号处理函数。

常见信号：

| 信号 | 含义 |
| --- | --- |
| `SIGINT` | 终端中断，通常来自 Ctrl-C |
| `SIGTERM` | 请求进程终止 |
| `SIGKILL` | 强制杀死，不能捕获 |
| `SIGSEGV` | 非法内存访问 |
| `SIGCHLD` | 子进程状态变化 |
| `SIGHUP` | 终端断开，常用于 reload |

信号处理函数里能做的事情非常有限，因为它可能在任意指令处打断程序。复杂逻辑通常应当交给主循环处理，信号处理函数只设置标志位或写入 self-pipe。

**Rust 对应：** 生产代码通常使用 `signal-hook`、Tokio signal 等封装，而不是手写裸 `sigaction`。

## I/O 模型

I/O 模型指：当程序发起 I/O（读文件、读 socket、写网络）时，如果数据还没准备好，**操作系统如何让你等待**、**如何通知你继续**。它不是“读写本身的 API”，而是“等待与唤醒的策略”。

从操作系统视角看，I/O 牵涉三个核心角色：

- **fd 与内核对象**：fd 指向内核里的文件/管道/socket。数据是否就绪由内核决定。
- **线程与调度器**：线程在等待时可能被挂起，CPU 让给别的线程。
- **事件通知机制**：内核在数据就绪时通知用户态，避免无意义轮询。

理解 I/O 模型的关键问题是：

1. 线程是在内核里睡眠，还是在用户态自旋轮询？
2. 同时管理很多连接时，是“一个连接一个线程”，还是“一个线程管理很多连接”？
3. 谁来通知“数据就绪”：内核主动通知，还是应用反复询问？

### 阻塞 I/O

调用 `read` 时，如果没有数据，线程会睡眠，直到数据到达。

优点是简单，缺点是大量连接会占用大量线程。

### 非阻塞 I/O

fd 设置成非阻塞后，如果没有数据，`read` 立即返回错误，例如 `EAGAIN`。程序需要稍后再试。

这意味着线程不会被内核挂起，但应用必须自己决定“什么时候再试”，通常会配合事件通知机制使用，否则就是忙等浪费 CPU。

非阻塞 I/O 通常和事件通知机制配合使用。

### I/O 多路复用

`select`、`poll`、`epoll`、`kqueue`、IOCP 等机制用于同时等待多个 I/O 事件。

Linux 高并发网络服务常用 `epoll`：

1. 把 fd 注册到 epoll。
2. 调用 `epoll_wait` 等待事件。
3. 哪些 fd 可读可写，就处理哪些 fd。

这里的核心价值是：**一个线程同时管理大量连接**，避免“一连接一线程”的爆炸式成本。

### 异步 I/O

异步 I/O 的核心思想是提交操作后不阻塞当前执行流，完成后再通知程序。不同平台实现差异很大，例如 Linux 的 io_uring、Windows 的 IOCP。

**Rust 对应：** Tokio、mio、async-std 等运行时把非阻塞 fd、事件循环、任务调度和 `Future` 组合起来，形成 Rust 的 async 生态。

### 一个对比表

| 模型 | 线程是否阻塞 | 谁负责等待 | 常见特点 |
| --- | --- | --- | --- |
| 阻塞 I/O | 是 | 内核 | 实现简单，但线程数与连接数近似 1:1 |
| 非阻塞 I/O | 否 | 应用 | 需要自行重试或配合事件通知 |
| 多路复用 | 取决于调用 `select`/`epoll_wait` 是否阻塞 | 内核 + 应用 | 一个线程等待多个 fd，适合高并发 |
| 异步 I/O | 否（提交后立即返回） | 内核 | 内核完成后通知，应用拿结果 |

如果你把“阻塞/非阻塞”理解为“单个 `read` 调用会不会让线程睡眠”，把“多路复用/异步 I/O”理解为“同时管理很多 I/O 的方式”，这四个概念就能串起来了。

----

## 第二部分：Rust 系统编程

这一部分开始使用 Rust。前面已经讲过 OS 概念，这里重点回答：Rust 的哪些类型和 API 对应哪些系统能力。

### Rust 系统编程的层次

| 层次 | 常用工具 | 特点 |
| --- | --- | --- |
| 标准库 | `std::fs`、`std::net`、`std::process`、`std::thread` | 安全、跨平台、够用优先 |
| 平台扩展 | `std::os::unix`、`std::os::windows` | 暴露平台特有能力 |
| 第三方封装 | `nix`、`rustix`、`socket2`、`memmap2` | 比 `libc` 更 Rust 化 |
| C ABI | `libc`、FFI | 能力完整，但 unsafe 多 |
| 裸机 / 内核 | `no_std`、内联汇编、启动代码 | 自己管理运行环境 |

实践原则：

- 能用标准库就先用标准库。
- 标准库不够时优先找成熟 crate。
- 必须碰底层 API 时，把 `unsafe` 包在小而清晰的边界内。
- 明确资源所有权，尤其是 fd、指针、映射内存和外部库返回的句柄。

----

## 文件 I/O

标准库的文件 I/O 是系统编程最常见入口。

```rust
use std::fs;

fn main() -> std::io::Result<()> {
    fs::write("hello.txt", "hello\n")?;
    let content = fs::read_to_string("hello.txt")?;
    print!("{content}");
    Ok(())
}
```

更细粒度的控制使用 `File` 和 `OpenOptions`：

```rust
use std::fs::OpenOptions;
use std::io::{Read, Seek, SeekFrom, Write};

fn main() -> std::io::Result<()> {
    let mut file = OpenOptions::new()
        .read(true)
        .write(true)
        .create(true)
        .truncate(false)
        .open("data.bin")?;

    file.write_all(b"hello")?;
    file.seek(SeekFrom::Start(0))?;

    let mut buf = [0; 5];
    file.read_exact(&mut buf)?;
    println!("{buf:?}");

    Ok(())
}
```

这里的 `?` 会把底层 I/O 错误作为 `std::io::Error` 传播。它可能对应权限不足、文件不存在、磁盘满、系统调用被中断等情况。

----

## 进程管理

Rust 用 `std::process::Command` 创建子进程。

```rust
use std::process::Command;

fn main() -> std::io::Result<()> {
    let output = Command::new("rustc")
        .arg("--version")
        .output()?;

    println!("{}", String::from_utf8_lossy(&output.stdout));
    Ok(())
}
```

带管道的例子：

```rust
use std::io::Write;
use std::process::{Command, Stdio};

fn main() -> std::io::Result<()> {
    let mut child = Command::new("findstr")
        .arg("error")
        .stdin(Stdio::piped())
        .stdout(Stdio::piped())
        .spawn()?;

    if let Some(mut stdin) = child.stdin.take() {
        stdin.write_all(b"ok\nerror: failed\n")?;
    }

    let output = child.wait_with_output()?;
    println!("{}", String::from_utf8_lossy(&output.stdout));
    Ok(())
}
```

在 Unix 上这背后通常涉及创建进程、设置 fd、执行新程序。在 Windows 上底层机制不同，但 `Command` 尽量提供统一抽象。

----

## 线程、Send 与 Sync

创建 OS 线程：

```rust
use std::thread;

fn main() {
    let data = vec![1, 2, 3];

    let handle = thread::spawn(move || {
        println!("{data:?}");
    });

    handle.join().unwrap();
}
```

`move` 把 `data` 的所有权移动进新线程，避免子线程引用主线程栈上的数据。

### Send

`Send` 表示一个类型的值可以安全地移动到另一个线程。

- `Vec<T>` 通常是 `Send`，前提是 `T: Send`。
- `Rc<T>` 不是 `Send`，因为引用计数不是原子的。
- `Arc<T>` 可以跨线程共享所有权。

### Sync

`Sync` 表示一个类型的共享引用 `&T` 可以安全地被多个线程同时使用。

- `Mutex<T>` 通常是 `Sync`，因为内部保证互斥访问。
- `RefCell<T>` 不是 `Sync`，因为运行时借用计数不是线程安全的。

```text
类型          Send    Sync
Rc<T>         否      否
Arc<T>        是      是
RefCell<T>    是*     否
Mutex<T>      是*     是*
```

`*` 表示通常还取决于内部的 `T` 是否满足相应约束。

----

## 同步原语

`Mutex` 用于互斥访问共享数据：

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = Vec::new();

    for _ in 0..4 {
        let counter = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            let mut value = counter.lock().unwrap();
            *value += 1;
        }));
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("{}", *counter.lock().unwrap());
}
```

`Arc` 负责跨线程共享所有权，`Mutex` 负责互斥访问内部值。

`RwLock` 适合读多写少：

```rust
use std::sync::RwLock;

fn main() {
    let value = RwLock::new(String::from("hello"));

    {
        let read = value.read().unwrap();
        println!("{read}");
    }

    {
        let mut write = value.write().unwrap();
        write.push_str(" world");
    }
}
```

注意：Rust 可以防止数据竞争，但仍然可能死锁。例如两个线程用相反顺序获取两把锁，类型系统不会自动替你发现。

----

## Unix 文件描述符

在 Unix 上，可以通过 `std::os::fd` 处理 fd。

```rust
use std::fs::File;
use std::os::fd::{AsRawFd, RawFd};

fn print_fd(file: &File) {
    let fd: RawFd = file.as_raw_fd();
    println!("fd = {fd}");
}

fn main() -> std::io::Result<()> {
    let file = File::open("Cargo.toml")?;
    print_fd(&file);
    Ok(())
}
```

几个重要类型：

| 类型 | 含义 |
| --- | --- |
| `RawFd` | 裸 fd，只是整数，不表达所有权 |
| `BorrowedFd<'a>` | 借用 fd，不负责关闭 |
| `OwnedFd` | 拥有 fd，`Drop` 时关闭 |

不要随便从 `RawFd` 构造 `File` 或 `OwnedFd`。如果所有权判断错了，可能重复关闭 fd，或者关闭了别人还在用的 fd。

----

## mmap 与内存映射

`mmap` 可以把文件或匿名内存映射到进程地址空间。常见用途：

- 大文件读取。
- 进程间共享内存。
- 自定义分配器。
- JIT 或加载器管理可执行内存。

使用 `memmap2` 的典型方式：

```rust
use memmap2::Mmap;
use std::fs::File;

fn main() -> std::io::Result<()> {
    let file = File::open("Cargo.toml")?;

    // SAFETY: 文件在映射期间保持打开，且这里只创建只读映射。
    let mmap = unsafe { Mmap::map(&file)? };

    let len = mmap.len().min(64);
    println!("{}", String::from_utf8_lossy(&mmap[..len]));
    Ok(())
}
```

为什么这里是 `unsafe`？因为内存映射绕过了普通 Rust 借用规则。文件可能被其他进程修改，映射区域也可能和外部状态相关。库无法完全证明这些访问总是安全。

----

## libc 与直接系统调用

当标准库没有暴露某个能力时，可以使用 `libc` 或更安全的封装 crate。

```rust
use std::ffi::CString;

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let path = CString::new("Cargo.toml")?;

    unsafe {
        let fd = libc::open(path.as_ptr(), libc::O_RDONLY);
        if fd < 0 {
            return Err(std::io::Error::last_os_error().into());
        }

        let mut buf = [0u8; 128];
        let n = libc::read(fd, buf.as_mut_ptr().cast(), buf.len());
        if n < 0 {
            let err = std::io::Error::last_os_error();
            libc::close(fd);
            return Err(err.into());
        }

        libc::close(fd);
        println!("{}", String::from_utf8_lossy(&buf[..n as usize]));
    }

    Ok(())
}
```

这类代码要特别注意：

- C 字符串必须以 `\0` 结尾，且内部不能有额外 NUL。
- 返回值要检查错误。
- fd、指针、分配内存必须有清晰释放路径。
- `unsafe` 块内的假设最好写成简短注释。

----

## FFI 调用 C

Rust 可以通过 C ABI 调用 C 函数，也可以把 Rust 函数导出给 C。

调用 C 函数：

```rust
unsafe extern "C" {
    fn puts(s: *const libc::c_char) -> libc::c_int;
}

fn main() {
    let message = std::ffi::CString::new("hello from C").unwrap();

    unsafe {
        puts(message.as_ptr());
    }
}
```

导出 Rust 函数：

```rust
#[no_mangle]
pub extern "C" fn rust_add(a: i32, b: i32) -> i32 {
    a + b
}
```

FFI 边界要格外明确：

- 用 `#[repr(C)]` 固定结构体布局。
- 不要让 Rust 引用、`String`、`Vec` 直接穿过不匹配的 ABI 边界。
- 谁分配，谁释放，规则要写清楚。
- panic 不应该跨过 FFI 边界。

----

## unsafe 与裸指针

`unsafe` 表示有些安全条件编译器无法证明，需要程序员负责。

常见 unsafe 场景：

| 场景 | 风险 |
| --- | --- |
| 解引用裸指针 | 指针可能为空、悬垂、未对齐 |
| 调用 C 函数 | ABI、生命周期、错误处理不受 Rust 检查 |
| 访问可变静态变量 | 可能产生数据竞争 |
| 实现 unsafe trait | 需要维护全局不变量 |
| 内联汇编 | 编译器无法理解副作用 |

裸指针示例：

```rust
fn main() {
    let mut value = 42;
    let ptr: *mut i32 = &mut value;

    unsafe {
        *ptr = 100;
    }

    println!("{value}");
}
```

写 unsafe 代码时，重点不是“让它能编译”，而是把不变量收束到最小范围：

- unsafe 块尽量短。
- 外层提供安全 API。
- 用类型表达所有权。
- 对复杂 unsafe 写清楚 `SAFETY:` 条件。

----

## no_std 开发

`#![no_std]` 表示不链接 Rust 标准库，只使用 `core`，通常用于内核、裸机、嵌入式等环境。

`std` 依赖操作系统能力，例如线程、文件、网络、堆分配、环境变量。没有操作系统时，这些能力不存在，或者需要你自己提供。

`no_std` 常见工作：

- 自己提供入口点。
- 自己处理 panic。
- 自己初始化内存和栈。
- 自己实现或接入分配器。
- 直接操作硬件寄存器或固件接口。

最小形式：

```rust
#![no_std]
#![no_main]

use core::panic::PanicInfo;

#[panic_handler]
fn panic(_info: &PanicInfo) -> ! {
    loop {}
}
```

如果需要堆分配，要实现全局分配器：

```rust
use core::alloc::{GlobalAlloc, Layout};

struct DummyAllocator;

unsafe impl GlobalAlloc for DummyAllocator {
    unsafe fn alloc(&self, _layout: Layout) -> *mut u8 {
        core::ptr::null_mut()
    }

    unsafe fn dealloc(&self, _ptr: *mut u8, _layout: Layout) {}
}
```

这部分已经接近 OS 内核或嵌入式开发，不再是普通用户态系统编程。

----

## 平台抽象

系统编程经常需要处理平台差异。Rust 用条件编译表达不同平台代码。

```rust
#[cfg(unix)]
fn platform_name() -> &'static str {
    "unix"
}

#[cfg(windows)]
fn platform_name() -> &'static str {
    "windows"
}

fn main() {
    println!("{}", platform_name());
}
```

Unix 特有能力通常在 `std::os::unix` 下，Windows 特有能力在 `std::os::windows` 下。

设计跨平台系统代码时，常见做法是：

- 业务逻辑保持平台无关。
- 把平台相关代码隔离在小模块里。
- 对外暴露统一 trait 或函数。
- 平台能力差异不能强行抹平时，就显式暴露差异。

----

## 异步编程

Rust 的 async 是语言级 `Future` 加运行时调度。

`Future` 本质上是一个可以被轮询的状态机：

```rust
use std::future::Future;
use std::pin::Pin;
use std::task::{Context, Poll};

struct PendingForever;

impl Future for PendingForever {
    type Output = ();

    fn poll(self: Pin<&mut Self>, _cx: &mut Context<'_>) -> Poll<()> {
        Poll::Pending
    }
}
```

实际应用通常使用 Tokio：

```rust
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    let task = tokio::spawn(async {
        sleep(Duration::from_millis(100)).await;
        println!("done");
    });

    task.await.unwrap();
}
```

把它和 OS I/O 模型联系起来看：

1. socket 设置为非阻塞。
2. 运行时把 fd 注册到 epoll/kqueue/IOCP。
3. `Future` 暂时无法完成时返回 `Poll::Pending`。
4. I/O 事件到达后，运行时通过 `Waker` 唤醒任务。
5. 任务再次被 poll，继续执行。

所以 async 不是魔法。它是在 OS 事件通知机制上构建的用户态任务调度。

----

## 调试与诊断

系统编程常见问题包括：

- 段错误。
- fd 泄漏。
- 死锁。
- 数据竞争。
- 内存泄漏。
- 系统调用失败。
- 性能被 I/O 或上下文切换拖垮。

常用工具：

| 工具 | 用途 |
| --- | --- |
| `RUST_BACKTRACE=1` | 打印 panic 回溯 |
| `strace` / `dtruss` | 跟踪系统调用 |
| `lsof` | 查看进程打开的文件和 socket |
| `perf` | Linux 性能分析 |
| `valgrind` | 内存错误和泄漏检查 |
| `gdb` / `lldb` | 原生调试 |
| `cargo flamegraph` | 火焰图 |
| `loom` | 并发模型测试 |
| `miri` | 检查部分 Rust 未定义行为 |

排查系统编程问题时，建议按层次定位：

1. Rust 层：所有权、生命周期、panic、错误传播是否正确。
2. 标准库层：是否正确处理 `Result`，是否泄漏资源。
3. OS 层：系统调用是否失败，fd、权限、路径、信号是否符合预期。
4. 硬件和性能层：缓存、上下文切换、内存映射、I/O 等待是否成为瓶颈。

----

## 学习路线建议

如果按顺序学，可以这样走：

1. 用户态 / 内核态，理解为什么需要系统调用。
2. 进程、线程和调度，理解程序如何运行。
3. 虚拟内存、分页、TLB，理解地址和内存安全的底层。
4. 文件描述符、管道、socket，理解“一切皆文件”。
5. 同步、IPC、信号，理解多任务协作。
6. Rust 标准库系统 API，先写安全的用户态程序。
7. Unix fd、`mmap`、`libc`、FFI，逐步进入底层。
8. `unsafe` 和 `no_std`，只在确实需要时深入。

第一部分解决“操作系统在做什么”，第二部分解决“Rust 如何安全地使用这些能力”。两条线分清楚之后，系统编程就不会再像一堆突然冒出来的名词。
