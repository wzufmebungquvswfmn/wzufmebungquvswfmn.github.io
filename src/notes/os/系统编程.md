# 操作系统及基础知识

快速跳转 >> [Rust 系统编程](#rust-系统编程)  

系统编程的主线是：

> 通过操作系统提供的接口，管理和控制计算机资源。

不同层次负责的事情：

1. **操作系统内核**管理 CPU、内存、设备等硬件资源。
2. **系统调用**是用户程序请求内核服务的唯一入口。
3. **并发模型**控制多任务如何共享 CPU。
4. **内存管理**决定进程如何使用物理和虚拟内存。
5. **文件与 I/O**是进程与外设交换数据的方式。
6. **进程间通信**让不同进程可以交换数据。

## 用户态与内核态

CPU 一般提供多个特权级别。以 Linux/x86 为例：

- **内核态**（ring 0）：可以执行任何指令（操作页表、中断、设备 I/O 等）。
- **用户态**（ring 3）：只能执行普通指令，访问受限的内存区域。

用户程序大部分时间在**用户态**运行。当需要内核服务（读文件、发网络包、分配内存）时，通过**系统调用**切换到内核态。

> 为什么要分两态？防止用户程序乱搞——比如一个 buggy 的进程直接修改了别的进程的内存，或者把系统搞崩。

## 系统调用

系统调用是用户程序请求内核服务的唯一入口。

常见分类：

| 类别 | 系统调用举例 |
| --- | --- |
| 进程控制 | `fork`、`execve`、`exit`、`waitpid` |
| 文件 I/O | `open`、`read`、`write`、`close`、`fsync` |
| 内存管理 | `mmap`、`munmap`、`brk` |
| 网络 | `socket`、`bind`、`listen`、`accept` |
| IPC | `pipe`、`shmget`、`semop` |
| 信号 | `kill`、`signal`、`sigaction` |

系统调用流程：

1. 用户程序把参数放到寄存器。
2. 触发 `syscall` / `int 0x80` 等指令。
3. CPU 切换到内核态。
4. 内核根据系统调用号执行对应函数。
5. 结果返回给用户程序，切换回用户态。

每次系统调用都有开销（上下文切换、TLB 刷新等），所以高并发场景下要尽量减少系统调用的次数。

> Rust 标准库已经封装了大部分系统调用，一般不需要直接写 `syscall` 指令。

### 系统调用 ABI

系统调用和普通函数调用用的不是同一套 ABI。以 Linux 为例：

- 系统调用号放在 `rax`（x86_64）或 `a7`（riscv64）。
- 参数放在 `rdi, rsi, rdx, r10, r8, r9`（x86_64）或 `a0~a5`（riscv64）。
- `syscall` 指令触发内核态切换，返回值在 `rax` / `a0`。
- 错误码通过返回值范围判断（如 x86_64 用 `-4095~-1` 表示错误）。

如果要在 Rust 中直接做系统调用（不走 libc），需要用 `core::arch::asm!` 内联汇编：

```rust
// x86_64 Linux 直接调用 write(1, buf, len)
fn sys_write(fd: u64, buf: *const u8, len: u64) -> i64 {
    let ret: i64;
    unsafe {
        core::arch::asm!(
            "syscall",
            in("rax") 1,       // 系统调用号：write=1
            in("rdi") fd,
            in("rsi") buf,
            in("rdx") len,
            out("rcx") _,     // syscall 会破坏 rcx
            out("r11") _,     // syscall 会破坏 r11
            lateout("rax") ret,
        );
    }
    ret
}
```

不同的 CPU 架构的系统调用 ABI 不同，移植时要注意：

| 架构 | 触发指令 | 系统调用号寄存器 | 参数寄存器 |
| --- | --- | --- | --- |
| x86_64 | `syscall` | `rax` | `rdi, rsi, rdx, r10, r8, r9` |
| aarch64 | `svc #0` | `x8` | `x0~x5` |
| riscv64 | `ecall` | `a7` | `a0~a5` |

## 进程与线程

### 进程

进程是资源分配的基本单位。

每个进程拥有：

- 独立的虚拟地址空间。
- 文件描述符表。
- 进程 ID（PID）。
- 环境变量、工作目录等。

进程间默认不共享内存。要交换数据需要用 IPC 机制。

```
┌─────────────────┐     ┌─────────────────┐
│  进程 A          │     │  进程 B          │
│  ┌───────────┐  │     │  ┌───────────┐  │
│  │ 地址空间   │  │     │  │ 地址空间   │  │
│  │ (独立)     │  │     │  │ (独立)     │  │
│  └───────────┘  │     │  └───────────┘  │
│  │ 文件描述符  │  │     │  │ 文件描述符  │  │
│  └───────────┘  │     │  └───────────┘  │
└─────────────────┘     └─────────────────┘
       │                       │
       └──────── IPC ──────────┘
```

### 线程

线程是 CPU 调度的基本单位。

同一进程内的线程：

- 共享地址空间和文件描述符。
- 各有独立的栈和寄存器上下文。
- 创建和切换开销比进程小。

```text
┌─────────────────────────────┐
│  进程                       │
│  ┌──────┐ ┌──────┐ ┌──────┐ │
│  │线程1 │ │线程2 │ │线程3 │ │
│  │(栈A) │ │(栈B) │ │(栈C) │ │
│  └──────┘ └──────┘ └──────┘ │
│  共享：地址空间、fd表        │
└─────────────────────────────┘
```

Rust 中创建线程：

```rust
use std::thread;

fn main() {
    let handle = thread::spawn(|| {
        println!("子线程");
    });

    println!("主线程");
    handle.join().unwrap(); // 等待子线程结束
}
```

### 进程 vs 线程

| 特性 | 进程 | 线程 |
| --- | --- | --- |
| 地址空间 | 独立 | 共享 |
| 资源开销 | 大 | 小 |
| 切换开销 | 大 | 小 |
| 通信 | IPC | 直接读写共享内存 |
| 隔离性 | 高（一个崩了不影响别的） | 低（一个崩了整个进程崩） |
| Rust 创建方式 | `std::process::Command` | `std::thread::spawn` |

### Send 和 Sync

Rust 的线程安全由两个标记 trait 静态保证：

- **`Send`**：类型可以安全地**转移**到另一个线程。几乎所有类型都实现了 `Send`，但 `Rc<T>` 没有（因为引用计数不是原子的）。
- **`Sync`**：类型可以安全地**共享引用**到另一个线程。即 `&T` 可以安全地在多个线程中访问。`Rc<T>` 和 `RefCell<T>` 没有实现 `Sync`。

```text
          Send（所有权可跨线程转移）     Sync（引用可跨线程共享）
Rc<T>     ❌                           ❌
Arc<T>    ✅                           ✅
RefCell<T> ✅                          ❌
Mutex<T>  ✅                           ✅
```

`thread::spawn` 的闭包必须是 `Send + 'static`——闭包捕获的数据必须能跨线程转移所有权，且不包含任何借用。

```rust
use std::thread;

fn main() {
    let data = vec![1, 2, 3];

    // move 强制闭包获取 data 的所有权，保证转移后数据有效
    let handle = thread::spawn(move || {
        println!("data: {:?}", data);
    });

    // 这里不能再使用 data，所有权已转移给线程
    handle.join().unwrap();
}
```

## 协程

协程是用户态并发，不由内核调度，而是由程序自己控制。

特点：

- 不依赖内核线程，轻量。
- 切换开销极小，只是保存/恢复寄存器。
- 一个线程可以跑成千上万个协程。
- 通常配合异步 I/O 使用。

> 线程是内核调度的最小单位，协程是用户态调度的最小单位。

Rust 中协程通过 `async/await` 实现，运行时如 tokio：

```rust
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    tokio::spawn(async {
        sleep(Duration::from_millis(100)).await;
        println!("协程1");
    });

    tokio::spawn(async {
        sleep(Duration::from_millis(50)).await;
        println!("协程2");
    });

    sleep(Duration::from_millis(200)).await;
}
```

### 上下文切换的原理

协程、绿色线程、甚至内核的线程切换，底层都是同一件事：**保存当前执行上下文，恢复另一个执行上下文**。

执行上下文包括：

- **被调用者保存的寄存器**（Callee-saved）：`s0~s11`（riscv64）、`rbx, rbp, r12~r15`（x86_64）。这些由当前函数负责保存。
- **栈指针 SP**：指向当前栈帧，切换 SP 就等于切换了栈。
- **返回地址 RA**：切换后从哪条指令继续执行。

手动上下文切换（riscv64）的核心汇编（简化）：

```text
# 保存当前上下文
sd   ra,  0(a0)    # 保存返回地址
sd   sp,  8(a0)    # 保存栈指针
sd   s0,  16(a0)   # 保存被调用者保存的寄存器
sd   s1,  24(a0)
# ... 继续保存 s2~s11

# 恢复目标上下文
ld   ra,  0(a1)    # 恢复返回地址
ld   sp,  8(a1)    # 恢复栈指针
ld   s0,  16(a1)   # 恢复被调用者保存的寄存器
ld   s1,  24(a1)
# ...
ret                 # 跳到目标上下文继续执行
```

> 协作式调度的关键是每个任务**主动 yield**，把 CPU 让给其他任务。因为没有时钟中断，如果某个任务不 yield，其他任务就永远得不到执行。

## 虚拟内存

虚拟内存是操作系统给每个进程提供的地址空间抽象。

核心好处：

- **隔离**：进程 A 不能直接访问进程 B 的内存。
- **简化**：每个进程认为自己在独占内存（地址从 0 到最大值）。
- **安全性**：用户程序无法访问内核空间。
- **按需加载**：只有真正用到的部分才被加载到物理内存。

典型 64 位 Linux 进程地址空间布局（简化）：

```text
0x0000_0000_0000_0000
┌─────────────────────┐
│  代码段 (.text)      │
├─────────────────────┤
│  数据段 (.data/.bss) │
├─────────────────────┤
│  堆                  │
│     ↓               │
│     空闲            │
│     ↑               │
│  栈                 │
├─────────────────────┤
│  环境变量 / 参数     │
├─────────────────────┤
│  内核空间（用户不可见）│
└─────────────────────┘
0xFFFF_FFFF_FFFF_FFFF
```

## 分页机制

虚拟内存通过**分页**实现。虚拟地址分成固定大小的**页**，物理内存分成同样大小的**页框**。

- 虚拟地址 → 页号 + 页内偏移。
- 页号通过**页表**查找到对应的物理页框号。
- 页内偏移直接对应物理页框内的偏移。

```text
虚拟地址：  [页号 | 偏移]
                 │
                 ▼
            ┌──────────┐
            │   页表    │
            │ vpn → ppn │
            └──────────┘
                 │
                 ▼
物理地址：  [物理页框号 | 偏移]
```

如果访问的页不在物理内存中，会触发**缺页中断**，由内核从磁盘加载。

Rust 中没有直接操作页表的 API，但可以通过 `mmap` 间接控制内存映射：

```rust
use std::alloc::{alloc, Layout};

fn main() {
    // Rust 默认走 jemalloc/glibc malloc
    // 底层通过 brk 或 mmap 分配虚拟内存
    let layout = Layout::new::<[u8; 4096]>();
    let ptr = unsafe { alloc(layout) };
    unsafe { *ptr = 42 };
    println!("分配了一个页: {}", unsafe { *ptr });
}
```

### SV39 三级页表

RISC-V 的 SV39 方案把 39 位虚拟地址分成三级索引，每级 9 位，最后 12 位是页内偏移：

```text
虚拟地址（39位有效）：
[VPN2:9位 | VPN1:9位 | VPN0:9位 | 页内偏移:12位]
    │          │          │
    ▼          ▼          ▼
  L2页表     L1页表     L0页表     → 物理页框
```

如果某个中间级页表的 PTE 设置了**大页标志**，则跳过后续级别。SV39 支持两种大页：

| 级别 | 页大小 | 有效虚拟地址位数 |
| --- | --- | --- |
| L0（标准页） | 4 KiB | 39 位 |
| L1（大页） | 2 MiB | 30 位 |
| L2（超大页） | 1 GiB | 21 位 |

### TLB 和 MMU

MMU（内存管理单元）负责把虚拟地址翻译成物理地址。TLB（ Translation Lookaside Buffer）是 MMU 内部的页表缓存。

- TLB 命中：直接拿到物理地址，无需访问内存页表。
- TLB 缺失：硬件遍历页表（或触发缺页异常），加载页表项到 TLB。
- TLB 刷新：修改页表后，需要使对应的 TLB 项失效，否则 MMU 会读到过期的映射。

TLB 刷新策略：

| 刷新粒度 | 触发时机 |
| --- | --- |
| 全部刷新 | 进程切换（切换地址空间） |
| 按页刷新 | `mprotect`、`munmap` 等单页操作 |
| 按 ASID 刷新 | 支持 ASID 的 CPU 可按进程刷新 |

## 文件描述符

在 Unix/Linux 中，**一切皆文件**。文件、socket、管道、设备等 I/O 资源，都通过文件描述符（fd）访问。

文件描述符是一个非负整数。每个进程有自己的 fd 表：

```text
fd 表：
 0 → 标准输入（stdin）
 1 → 标准输出（stdout）
 2 → 标准错误（stderr）
 3 → 打开的文件 / socket / 等
```

Rust 中操作文件描述符：

```rust
use std::fs::File;
use std::os::unix::io::{RawFd, AsRawFd};

fn main() {
    let file = File::open("Cargo.toml").unwrap();
    let fd: RawFd = file.as_raw_fd();
    println!("文件描述符: {fd}");
}
```

## 进程间通信

进程间通信（IPC）的方式：

| 方式 | 特点 | Rust 中使用 |
| --- | --- | --- |
| 管道 | 单向通信，适合父子进程 | `std::process::Command` 的管道 |
| 命名管道（FIFO） | 不相关进程也可用 | 通过文件系统创建 |
| 信号量 | 同步/互斥 | 通常是 `extern crate` 或系统调用 |
| 共享内存 | 最快，需配合同步 | `mmap` + `MAP_SHARED` |
| 消息队列 | 消息传递 | `libc` 的 `mq_*` 系列 |
| 信号 | 异步通知 | 见[信号](#信号) |
| Unix Socket | 本地双工通信 | `std::os::unix::net::UnixStream` |
| TCP/UDP | 远程通信 | `std::net::TcpStream` |

### 管道示例

```rust
use std::process::{Command, Stdio};
use std::io::Read;

fn main() {
    // 等价于 echo hello | wc -c
    let echo = Command::new("echo")
        .arg("hello")
        .stdout(Stdio::piped())
        .spawn()
        .unwrap();

    let output = Command::new("wc")
        .arg("-c")
        .stdin(echo.stdout.unwrap())
        .output()
        .unwrap();

    println!("结果: {}", String::from_utf8_lossy(&output.stdout));
}
```

### Unix Socket 示例

```rust
use std::os::unix::net::UnixListener;
use std::thread;

fn main() {
    let socket_path = "/tmp/example.sock";

    // 服务端
    let listener = UnixListener::bind(socket_path).unwrap();
    thread::spawn(move || {
        let (mut stream, _) = listener.accept().unwrap();
        use std::io::Read;
        let mut buf = [0; 1024];
        let n = stream.read(&mut buf).unwrap();
        println!("收到: {}", String::from_utf8_lossy(&buf[..n]));
    });

    // 客户端
    use std::os::unix::net::UnixStream;
    use std::io::Write;
    let mut stream = UnixStream::connect(socket_path).unwrap();
    stream.write_all(b"hello via unix socket").unwrap();
}
```

### mpsc 通道

除了管道和 Unix Socket，Rust 标准库还提供了 **mpsc（多生产者单消费者）** 通道用于线程间通信：

```rust
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    // 多生产者：克隆发送端
    let tx2 = tx.clone();

    thread::spawn(move || {
        tx.send("来自线程1").unwrap();
    });

    thread::spawn(move || {
        tx2.send("来自线程2").unwrap();
    });

    // 单消费者：按发送顺序接收
    for msg in rx {
        println!("收到: {msg}");
    }
}
```

特点：

- 发送端可以克隆（`clone`），任意数量的生产者往同一个通道发数据。
- 接收端只有一个，按发送顺序取出消息。
- 发送和接收是不同步的（发送不会阻塞等待接收），通道内部有缓冲区。
- 所有发送端关闭后，接收端的迭代器自然结束。

`mpsc` 适合**一个消费者处理多个生产者**的通信模式。如果需要多消费者，考虑使用第三方库（如 `crossbeam`）或异步通道（`tokio::sync::mpsc`）。

## 同步机制

多线程/多进程访问共享资源时，需要同步来避免数据竞争。

### 互斥锁（Mutex）

同一时刻只有一个线程能持有锁：

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            let mut num = counter.lock().unwrap();
            *num += 1;
        }));
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("结果: {}", *counter.lock().unwrap());
}
```

`Arc<T>`（原子引用计数）和 `Mutex<T>` 经常搭配使用：`Arc` 负责跨线程共享所有权，`Mutex` 负责互斥访问。单独用 `Arc` 只能共享不可变数据，单独用 `Mutex` 无法跨线程传递所有权（需要 `Arc`）。

### 读写锁（RwLock）

多个读者可以同时读，写者独占：

```rust
use std::sync::RwLock;

fn main() {
    let data = RwLock::new(42);

    // 多个读者可以同时读
    let read_guard = data.read().unwrap();
    println!("读到: {read_guard}");
    drop(read_guard);

    // 写者独占
    let mut write_guard = data.write().unwrap();
    *write_guard += 1;
    drop(write_guard);
}
```

### 条件变量（Condvar）

线程等待某个条件成立：

```rust
use std::sync::{Arc, Condvar, Mutex};
use std::thread;

fn main() {
    let pair = Arc::new((Mutex::new(false), Condvar::new()));
    let pair2 = Arc::clone(&pair);

    thread::spawn(move || {
        let (lock, cvar) = &*pair2;
        let mut ready = lock.lock().unwrap();
        *ready = true;
        cvar.notify_one(); // 通知等待的线程
    });

    let (lock, cvar) = &*pair;
    let mut ready = lock.lock().unwrap();
    while !*ready {
        ready = cvar.wait(ready).unwrap(); // 等待条件成立
    }
    println!("条件已满足");
}
```

### 原子操作

对于简单的整数操作，原子操作比锁轻量得多：

```rust
use std::sync::atomic::{AtomicUsize, Ordering};

fn main() {
    let counter = AtomicUsize::new(0);

    // 原子加 1
    counter.fetch_add(1, Ordering::SeqCst);
    println!("{}", counter.load(Ordering::SeqCst));
}
```

### 内存序（Memory Ordering）

原子操作的内存序决定了**多核之间能观察到什么顺序**。Rust 提供了三种主要的内存序：

| 内存序 | 保证 | 代价 |
| --- | --- | --- |
| `Relaxed` | 仅保证原子性，不保证顺序 | 最低（无编译器/CPU 重排序限制） |
| `Release` / `Acquire` | 写用 Release、读用 Acquire，形成**happens-before**关系 | 中等 |
| `SeqCst` | 全局一致顺序，所有核看到相同的操作顺序 | 最高（可能触发全屏障指令） |

```rust
use std::sync::atomic::{AtomicBool, Ordering};

// 手写 OnceCell：利用 Acquire-Release 保证初始化完成可见性
static INITIALIZED: AtomicBool = AtomicBool::new(false);
static mut DATA: u64 = 0;

fn init() {
    // 只初始化一次
    if !INITIALIZED.load(Ordering::Acquire) {
        unsafe { DATA = 42 };            // 数据写
        INITIALIZED.store(true, Ordering::Release); // 释放屏障
    }
}
```

### 自旋锁（Spinlock）

自旋锁用 `compare_exchange_weak` 忙等（busy-wait），适合短临界区：

```rust
use std::sync::atomic::{AtomicBool, Ordering};
use std::hint::spin_loop;

pub struct Spinlock {
    locked: AtomicBool,
}

impl Spinlock {
    pub fn new() -> Self { Spinlock { locked: AtomicBool::new(false) } }

    pub fn lock(&self) {
        // CAS 循环：期待 locked==false，尝试设为 true
        while self.locked.compare_exchange_weak(
            false, true,
            Ordering::Acquire,
            Ordering::Relaxed,
        ).is_err() {
            spin_loop(); // 提示 CPU：我在自旋等锁
        }
    }

    pub fn unlock(&self) {
        self.locked.store(false, Ordering::Release);
    }
}
```

### RAII 守卫（Guard）

RAII（Resource Acquisition Is Initialization）是 Rust 管理资源的核心理念——**资源在构造时获取，在析构时释放**。

用自旋锁配上 RAII Guard，就能实现作用域结束自动解锁：

```rust
use std::ops::{Deref, DerefMut};

// Spinlock 的 RAII 守卫
pub struct SpinlockGuard<'a, T> {
    lock: &'a Spinlock,
    data: &'a mut T,
}

impl<'a, T> Deref for SpinlockGuard<'a, T> {
    type Target = T;
    fn deref(&self) -> &T { self.data }
}

impl<'a, T> DerefMut for SpinlockGuard<'a, T> {
    fn deref_mut(&mut self) -> &mut T { self.data }
}

impl<'a, T> Drop for SpinlockGuard<'a, T> {
    fn drop(&mut self) {
        self.lock.unlock(); // 离开作用域自动解锁！
    }
}
```

> 这就是标准库 `Mutex::lock()` 返回 `MutexGuard` 的原理——Guard 离开作用域自动 `Drop`，在 `drop` 里释放锁。你永远不用担心忘记解锁。

## 信号

信号是操作系统用来异步通知进程的机制。

常见信号：

| 信号 | 默认行为 | 常见来源 |

        }

        println!("工作中...");

    }
}

## 操作文件描述符

Unix 下可以直接操作原始 fd：

```rust
use std::fs::File;
use std::os::unix::io::{FromRawFd, RawFd, AsRawFd};
use std::io::Read;

fn main() -> std::io::Result<()> {
    // 获取 fd
    let file = File::open("Cargo.toml")?;
    let fd: RawFd = file.as_raw_fd();
    println!("fd 是: {fd}");

    // 从原始 fd 重建 File（转移所有权）
    let mut file2 = unsafe { File::from_raw_fd(fd) };
    let mut content = String::new();
    file2.read_to_string(&mut content)?;
    println!("前100字符: {}", &content[..100.min(content.len())]);

    Ok(())
}
```

> 注意 `from_raw_fd` 是 unsafe 的，因为 Rust 无法验证这个 fd 是否有效、是否已被其他地方使用。

### 文件描述符表

内核为每个进程维护一张**文件描述符表**，每个表项指向一个打开的文件/socket 等。

```text
fd 表（数组/向量）:
索引: 0    1    2    3    4    5    6
     ┌────┬────┬────┬────┬────┬────┬────┐
     │stdin│stdout│stderr│  /a.txt  │  ...  │
     └────┴────┴────┴────┴────┴────┴────┘
```

关键设计点：

- **fd 复用**：新打开的 fd 取当前**最小可用的编号**（关闭 3 后，下一个打开的文件用 3）。
- **动态分发**：fd 表项可以是 `Arc<dyn File>`，统一抽象文件、socket、管道等。
- **引用计数**：多个 fd 可以指向同一个内核文件对象（如 `dup` 复制 fd），`Arc` 负责自动释放。

实现一个简单的 fd 表：

```rust
use std::sync::Arc;

// 假设的 File trait
pub trait File: Send + Sync {
    fn read(&self, buf: &mut [u8]) -> usize;
    fn write(&self, buf: &[u8]) -> usize;
}

pub struct FdTable {
    // Option 表示槽位空闲
    entries: Vec<Option<Arc<dyn File>>>,
}

impl FdTable {
    pub fn alloc(&mut self, file: Arc<dyn File>) -> usize {
        // 查找最小可用 fd
        for (i, entry) in self.entries.iter_mut().enumerate() {
            if entry.is_none() {
                *entry = Some(file);
                return i;
            }
        }
        // 没有空闲槽，追加到末尾
        let fd = self.entries.len();
        self.entries.push(Some(file));
        fd
    }

    pub fn free(&mut self, fd: usize) {
        if let Some(entry) = self.entries.get_mut(fd) {
            *entry = None; // Arc drop 自动关闭文件
        }
    }
}
```

> 标准库的 `File` 底层就是通过 fd 表管理的。Rust 的 `OwnedFd` 类型（std 1.63+）提供了类型安全的 fd 所有权。

## 裸指针与不安全

系统编程经常需要突破 Rust 的安全限制——比如**自己管理内存**、**类型双关**、**操作硬件寄存器**。

```rust
fn main() {
    let mut x = 42;
    let raw_ptr: *mut i32 = &mut x;

    unsafe {
        *raw_ptr = 100;
        println!("通过裸指针修改: {x}"); // 输出 100

        // 手工分配和释放内存
        let layout = std::alloc::Layout::new::<[u8; 1024]>();
        let mem = std::alloc::alloc(layout);
        *(mem as *mut u32) = 0xDEAD_BEEF;
        println!("手工分配的内存: {:x}", *(mem as *mut u32));
        std::alloc::dealloc(mem, layout);
    }
}
```

用 `unsafe` 的常见场景：

| 场景 | 说明 |
| --- | --- |
| 解引用裸指针 | Rust 无法保证指针有效 |
| 调用 unsafe 函数 | 如 `std::mem::transmute` |
| 访问可变全局变量 | 可能造成数据竞争 |
| 实现 unsafe trait | 如 `Send`、`Sync` |
| FFI 调用 | 调用 C 函数 |

> 记住：`unsafe` 不是关闭安全检查，而是**由你来保证安全**。

## no_std 开发

在 OS 内核或裸机环境中，没有标准库可用（`#![no_std]`）。一切基础组件都要自己实现。

> 这是通往 OS 内核开发的必经之路——没有标准库，一切自己动手。

### 内存原语

标准库的 `memcpy`、`memset` 等内存操作在 no_std 下需要自己实现：

```rust
// 手写 memcpy（逐字节拷贝）
#[no_mangle]
pub unsafe extern "C" fn memcpy(dest: *mut u8, src: *const u8, n: usize) -> *mut u8 {
    for i in 0..n {
        *dest.add(i) = *src.add(i);
    }
    dest
}

// 手写 memset
#[no_mangle]
pub unsafe extern "C" fn memset(s: *mut u8, c: i32, n: usize) -> *mut u8 {
    for i in 0..n {
        *s.add(i) = c as u8;
    }
    s
}
```

### 分配器

`#[global_allocator]` 让自定义分配器无缝替换默认的 jemalloc：

```rust
use core::alloc::{GlobalAlloc, Layout};

// Bump 分配器——简单快速，但不能释放单个对象
pub struct BumpAllocator {
    heap_start: usize,
    heap_end: usize,
    next: core::sync::atomic::AtomicUsize,
}

unsafe impl GlobalAlloc for BumpAllocator {
    unsafe fn alloc(&self, layout: Layout) -> *mut u8 {
        loop {
            let current = self.next.load(core::sync::atomic::Ordering::Relaxed);
            let aligned = (current + layout.align() - 1) & !(layout.align() - 1);
            let new_next = aligned + layout.size();
            if new_next > self.heap_end {
                return core::ptr::null_mut(); // OOM
            }
            // CAS 保证线程安全
            if self.next.compare_exchange_weak(
                current, new_next,
                core::sync::atomic::Ordering::SeqCst,
                core::sync::atomic::Ordering::Relaxed,
            ).is_ok() {
                return aligned as *mut u8;
            }
        }
    }

    unsafe fn dealloc(&self, _ptr: *mut u8, _layout: Layout) {
        // Bump 分配器不支持单对象释放
    }
}
```

### 分配器策略对比

| 策略 | 分配速度 | 内存利用率 | 碎片 | 适用场景 |
| --- | :---: | :---: | :---: | --- |
| **Bump**（线性分配） | ⚡ 最快（O(1)） | 低（不能释放） | 无 | 临时分配、一次性分配 |
| **Free-list**（空闲链表） | 慢（O(n) 遍历） | 高（可重用） | 有（外部碎片） | 通用场景 |
| **Buddy**（伙伴系统） | 中（O(log n)） | 中 | 内部碎片 | 内核页分配 |

- **Bump 分配器**：维护一个 `next` 指针，分配时把指针往上移。不能释放单个对象，只能整体重置。适合短生命周期的临时分配（如帧分配器）。
- **Free-list 分配器**：把空闲内存块用**侵入式链表**串起来（链表节点直接存在空闲内存中），分配时遍历找 first-fit 或 best-fit 的块。释放时会尝试合并相邻空闲块。
- **线程安全**：no_std 下没有 `std::sync::Mutex`，通常用原子 CAS 实现（如 Bump 分配器中的 `compare_exchange_weak`），或者强制单线程使用。

## FFI 调用 C 代码

Rust 可以通过 FFI 直接调用 C 动态库。

### 调用 libc 的随机数函数

```rust
use std::ffi::CString;

extern "C" {
    fn rand() -> i32;
    fn puts(s: *const i8) -> i32;
}

fn main() {
    unsafe {
        let n = rand();
        println!("随机数: {n}");

        let s = CString::new("hello from C").unwrap();
        puts(s.as_ptr());
    }
}
```

### 链接自定义 C 库

假设有 C 代码：

```c
// math_helper.c
int add(int a, int b) {
    return a + b;
}
```

编译为静态库并链接：

```rust
// build.rs
fn main() {
    println!("cargo:rustc-link-lib=static=math_helper");
    println!("cargo:rerun-if-changed=math_helper.c");
}
```

Rust 端：

```rust
extern "C" {
    fn add(a: i32, b: i32) -> i32;
}

fn main() {
    unsafe {
        println!("3 + 4 = {}", add(3, 4));
    }
}
```

### 从 C 调用 Rust

```rust
// 导出为 C 函数
#[no_mangle]
pub extern "C" fn rust_add(a: i32, b: i32) -> i32 {
    a + b
}

// 生成动态库，C 端可以直接调用
// cargo build --release 生成 .so/.dll/.dylib
```

> FFI 是目前 Rust 系统编程中必不可少的技能。很多操作系统 API 只有 C 接口，Rust 需要通过 FFI 调用。

## libc 直接调用系统调用

通过 `libc` crate，可以直接调用 POSIX API：

```rust
use std::ffi::CString;

fn main() -> Result<(), Box<dyn std::error::Error>> {
    // 直接 open/read/write/close
    unsafe {
        let path = CString::new("Cargo.toml")?;
        let fd = libc::open(path.as_ptr(), libc::O_RDONLY);
        if fd < 0 {
            return Err("open failed".into());
        }

        let mut buf = [0u8; 1024];
        let n = libc::read(fd, buf.as_mut_ptr() as *mut libc::c_void, 1024);
        println!("读了 {} 字节: {:?}", n, String::from_utf8_lossy(&buf[..n as usize]));

        libc::close(fd);
    }
    Ok(())
}
```

常见 libc 封装：

| 系统调用 | libc 函数 | Rust 标准库封装 |
| --- | --- | --- |
| `open` | `libc::open` | `File::open` |
| `read`/`write` | `libc::read`/`libc::write` | `Read`/`Write` trait |
| `mmap` | `libc::mmap` | `memmap2` |
| `fork` | `libc::fork` | `std::process::Command` |
| `pipe` | `libc::pipe` | `std::process` 管道 |
| `ioctl` | `libc::ioctl` | 无，需 FFI |
| `socket` | `libc::socket` | `std::net::TcpStream` |

## 异步编程

Rust 的异步编程通过 `async/await` 实现，核心是 `Future` trait 和 `Waker` 唤醒机制。

```rust
use std::future::Future;
use std::pin::Pin;
use std::task::{Context, Poll};

// 一个简单的 Future：等待指定次数后完成
struct WaitForever;

impl Future for WaitForever {
    type Output = &'static str;

    fn poll(self: Pin<&mut Self>, _cx: &mut Context<'_>) -> Poll<Self::Output> {
        Poll::Pending // 永远 pending
    }
}
```

实际开发中通常使用 tokio 运行时：

```rust
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    let handle = tokio::spawn(async {
        sleep(Duration::from_millis(100)).await;
        println!("异步任务完成");
    });

    // 等待所有任务
    handle.await.unwrap();
}
```

### Future 与 Waker

Rust 的 Future 是**可轮询的状态机**，核心 trait 只有两个方法：

```rust
pub trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}

pub enum Poll<T> {
    Ready(T),
    Pending,
}
```

- `poll` 返回 `Poll::Ready(val)` 表示计算完成，返回 `Poll::Pending` 表示还没好。
- `Context` 中包含 `Waker`，异步任务在 I/O 就绪后通过 `waker.wake()` 通知执行器**再来轮询一次**。

```rust
use std::future::Future;
use std::pin::Pin;
use std::task::{Context, Poll, Waker};
use std::sync::{Arc, Mutex};

// 手动实现一个可等待的 Timer Future
struct TimerFuture {
    shared: Arc<Mutex<SharedState>>,
}

struct SharedState {
    completed: bool,
    waker: Option<Waker>,
}

impl Future for TimerFuture {
    type Output = ();
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<()> {
        let mut shared = self.shared.lock().unwrap();
        if shared.completed {
            Poll::Ready(())
        } else {
            // 保存 waker，等时间到了通过它唤醒
            shared.waker = Some(cx.waker().clone());
            Poll::Pending
        }
    }
}
```

### tokio::select!

`select!` 宏可以同时等待多个 Future，谁先完成就处理谁：

```rust
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    tokio::pin! {
        let slow = sleep(Duration::from_secs(5));
        let fast = sleep(Duration::from_secs(1));
    }

    tokio::select! {
        _ = &mut slow => println!("慢的完成了"),
        _ = &mut fast => println!("快的先完成了"),
    }

    // 输出 "快的先完成了"，且 slow 会被丢弃（取消）
}
```

常见用法：

- **超时控制**：`select! { res = op() => ..., _ = sleep(timeout) => ... }`
- **竞速**：多个数据源同时请求，谁先返回用谁的。
- **优雅关闭**：监听一个 `shutdown` 信号，收到后退出循环。

> Rust 的 async 是零成本抽象——没有隐式的运行时堆分配，没有隐式的任务调度。你只为你使用的功能付费。
