# 常用库

Rust 项目里经常会用一些生态库来补齐标准库没有直接提供的能力，比如错误处理、序列化、HTTP 请求、命令行参数解析、日志追踪等。

## 速览

| 库或模块 | 主要用途 | 面试时一句话 |
| --- | --- | --- |
| [anyhow](#anyhow) | 应用层错误处理 | 适合在二进制程序里快速传播各种错误 |
| [thiserror](#thiserror) | 自定义错误类型 | 适合在库代码或业务边界定义结构化错误 |
| [serde](#serde) | 序列化/反序列化框架 | Rust 生态里最常用的数据转换框架 |
| [serde_json](#serde_json) | JSON 读写 | 基于 serde，把结构体和 JSON 互转 |
| [toml](#toml) | TOML 配置解析 | 常用于读取配置文件，例如 Cargo.toml 风格配置 |
| [bytes](#bytes) | 高效字节缓冲区 | 网络编程里常用，减少字节数据拷贝 |
| [reqwest](#reqwest) | HTTP 客户端 | 用来发 HTTP 请求，支持 async、JSON、header 等 |
| [clap](#clap) | 命令行参数解析 | 用 derive 宏快速定义 CLI 参数 |
| [tracing](#tracing) | 结构化日志和链路追踪 | 比 println!/log 更适合异步和服务端项目 |
| [tracing-subscriber](#tracing-subscriber) | 日志收集和格式化 | 负责初始化 tracing 的输出格式和过滤级别 |

## anyhow

`anyhow` 用来简化应用层错误处理。它的核心类型是 `anyhow::Result<T>`，本质上等价于 `Result<T, anyhow::Error>`。

适合用在：

- CLI 工具。
- Web 服务的内部启动流程。
- demo、脚本、业务应用。
- 不想为每一种错误都定义 enum 的地方。

不太适合用在：

- 公共库的 API 边界。
- 需要让调用方精确匹配错误类型的地方。

### 基本用法

```rust
use anyhow::{Context, Result};

fn read_config(path: &str) -> Result<String> {
    let content = std::fs::read_to_string(path)
        .with_context(|| format!("读取配置文件失败: {path}"))?;

    Ok(content)
}
```

关键点：

- `?` 可以自动把不同错误转换成 `anyhow::Error`。
- `Context` 可以给错误补充上下文。
- 面试时可以说：`anyhow` 更偏应用层，重点是方便地传播和包装错误。

## thiserror

`thiserror` 用来定义自己的错误类型。它通过 `#[derive(Error)]` 自动实现 `std::error::Error` 和 `Display`。

适合用在：

- library crate。
- 业务边界清晰的模块。
- 希望调用方能 `match` 不同错误分支的场景。

### 基本用法

```rust
use thiserror::Error;

#[derive(Debug, Error)]
pub enum ConfigError {
    #[error("配置文件不存在: {0}")]
    NotFound(String),

    #[error("配置格式错误")]
    InvalidFormat,

    #[error("IO 错误")]
    Io(#[from] std::io::Error),
}

pub type Result<T> = std::result::Result<T, ConfigError>;
```

关键点：

- `#[error("...")]` 定义错误展示文本。
- `#[from]` 会自动实现 `From`，方便 `?` 转换错误。
- 面试时可以说：`thiserror` 负责“定义错误类型”，`anyhow` 负责“在应用层承接和传播错误”。

## serde

`serde` 是 Rust 生态中最常用的序列化和反序列化框架。它本身提供通用 trait，具体格式由 `serde_json`、`toml`、`bincode` 等库实现。

常用 derive：

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Serialize, Deserialize)]
struct User {
    id: u64,
    name: String,
}
```

常用属性：

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Serialize, Deserialize)]
struct Config {
    #[serde(rename = "server_port")]
    port: u16,

    #[serde(default)]
    debug: bool,

    #[serde(skip_serializing_if = "Option::is_none")]
    token: Option<String>,
}
```

关键点：

- `Serialize` 表示 Rust 类型可以转成外部格式。
- `Deserialize` 表示外部格式可以转回 Rust 类型。
- `rename` 用来处理字段名不一致。
- `default` 用来给缺失字段填默认值。
- 面试时可以说：serde 的优势是类型安全、性能好、格式无关。

## serde_json

`serde_json` 是 serde 的 JSON 格式实现，用来把 Rust 结构体和 JSON 字符串互转。

### 结构体转 JSON

```rust
use serde::Serialize;

#[derive(Serialize)]
struct User {
    id: u64,
    name: String,
}

fn main() -> serde_json::Result<()> {
    let user = User {
        id: 1,
        name: "Alice".to_string(),
    };

    let json = serde_json::to_string(&user)?;
    println!("{json}");

    Ok(())
}
```

### JSON 转结构体

```rust
use serde::Deserialize;

#[derive(Debug, Deserialize)]
struct User {
    id: u64,
    name: String,
}

fn main() -> serde_json::Result<()> {
    let text = r#"{"id":1,"name":"Alice"}"#;
    let user: User = serde_json::from_str(text)?;

    println!("{user:?}");

    Ok(())
}
```

### 动态 JSON

```rust
let value: serde_json::Value = serde_json::from_str(r#"{"name":"Alice"}"#)?;
println!("{}", value["name"]);
```

关键点：

- 能定义结构体时优先用结构体，不要到处用 `Value`。
- `Value` 适合字段不固定、临时处理、调试。
- 面试时可以说：`serde_json` 常用于 Web API 的请求体和响应体处理。

## toml

`toml` 用来读取 TOML 格式配置文件。Rust 项目里的 `Cargo.toml` 就是 TOML 格式。

示例配置：

```toml
[server]
host = "127.0.0.1"
port = 8080
```

解析代码：

```rust
use serde::Deserialize;

#[derive(Debug, Deserialize)]
struct AppConfig {
    server: ServerConfig,
}

#[derive(Debug, Deserialize)]
struct ServerConfig {
    host: String,
    port: u16,
}

fn load_config(text: &str) -> toml::de::Result<AppConfig> {
    toml::from_str(text)
}
```

关键点：

- TOML 适合人手写配置。
- 配合 `serde::Deserialize` 可以直接转成结构体。
- 面试时可以说：配置文件解析一般会用 `std::fs` 读取文本，再用 `toml::from_str` 反序列化。

## bytes

`bytes` 提供高效字节缓冲区，常见类型是 `Bytes` 和 `BytesMut`。

它常出现在：

- 网络协议处理。
- HTTP body。
- Tokio 生态。
- 需要频繁切分、复制字节数据的地方。

### Bytes

`Bytes` 是不可变字节缓冲区，可以便宜地 clone。clone 时通常共享底层内存，而不是立刻复制全部字节。

```rust
use bytes::Bytes;

let data = Bytes::from_static(b"hello");
let cloned = data.clone();

assert_eq!(&data[..], b"hello");
assert_eq!(&cloned[..], b"hello");
```

### BytesMut

`BytesMut` 是可变缓冲区，适合一点点写入数据。

```rust
use bytes::{BufMut, BytesMut};

let mut buf = BytesMut::with_capacity(64);
buf.put_u8(1);
buf.put_slice(b"hello");

let frozen = buf.freeze();
```

关键点：

- `Bytes` 偏只读共享。
- `BytesMut` 偏可变构建。
- `freeze()` 可以把 `BytesMut` 转成 `Bytes`。
- 面试时可以说：`bytes` 常用于网络 I/O，核心价值是减少不必要的内存拷贝。

## reqwest

`reqwest` 是 Rust 常用 HTTP 客户端，支持异步请求、JSON、header、timeout 等。

### GET 请求

```rust
use anyhow::Result;

#[tokio::main]
async fn main() -> Result<()> {
    let text = reqwest::get("https://example.com")
        .await?
        .text()
        .await?;

    println!("{text}");

    Ok(())
}
```

### POST JSON

```rust
use anyhow::Result;
use serde::{Deserialize, Serialize};

#[derive(Serialize)]
struct LoginRequest {
    username: String,
    password: String,
}

#[derive(Debug, Deserialize)]
struct LoginResponse {
    token: String,
}

async fn login() -> Result<LoginResponse> {
    let client = reqwest::Client::new();

    let response = client
        .post("https://api.example.com/login")
        .json(&LoginRequest {
            username: "alice".to_string(),
            password: "secret".to_string(),
        })
        .send()
        .await?
        .error_for_status()?
        .json::<LoginResponse>()
        .await?;

    Ok(response)
}
```

关键点：

- 简单请求可以用 `reqwest::get`。
- 复杂请求建议复用 `reqwest::Client`。
- `.json(&value)` 依赖 `serde::Serialize`。
- `.json::<T>()` 依赖 `serde::Deserialize`。
- `.error_for_status()` 可以把 4xx/5xx 状态码转成错误。

## clap

`clap` 用来解析命令行参数。现在最常用的是 derive 写法。

```rust
use clap::Parser;

#[derive(Debug, Parser)]
#[command(name = "demo")]
#[command(about = "一个命令行工具示例")]
struct Args {
    #[arg(short, long)]
    config: Option<String>,

    #[arg(short, long, default_value_t = 8080)]
    port: u16,

    #[arg(long)]
    debug: bool,
}

fn main() {
    let args = Args::parse();
    println!("{args:?}");
}
```

运行示例：

```bash
cargo run -- --config app.toml --port 3000 --debug
```

关键点：

- `Parser` 用来解析整个命令行参数。
- `#[arg(short, long)]` 表示同时支持 `-c` 和 `--config`。
- `default_value_t` 设置默认值。
- `bool` 字段常作为开关参数。

## tracing

`tracing` 用来做结构化日志和链路追踪。相比 `println!`，它能记录日志级别、字段、上下文 span，更适合异步和服务端项目。

### 基本日志

```rust
use tracing::{debug, error, info, warn};

fn main() {
    info!("服务启动");
    warn!("配置项缺失，使用默认值");
    debug!(port = 8080, "监听端口");
    error!("请求处理失败");
}
```

### span

`span` 表示一段执行上下文，例如一次请求、一个任务、一次数据库查询。

```rust
use tracing::{info, instrument};

#[instrument]
async fn handle_request(user_id: u64) {
    info!("开始处理请求");
}
```

关键点：

- `info!`、`warn!`、`error!` 等宏用于输出不同级别日志。
- 字段写法如 `user_id = 1`，便于机器解析。
- `#[instrument]` 可以自动记录函数参数和调用上下文。

## tracing-subscriber

`tracing-subscriber` 负责接收、过滤、格式化并输出 `tracing` 产生的事件。

最简单初始化：

```rust
fn main() {
    tracing_subscriber::fmt::init();

    tracing::info!("hello tracing");
}
```

带日志级别：

```rust
use tracing_subscriber::EnvFilter;

fn main() {
    tracing_subscriber::fmt()
        .with_env_filter(EnvFilter::from_default_env())
        .init();

    tracing::info!("服务启动");
}
```

运行时设置日志级别：

```bash
RUST_LOG=info cargo run
RUST_LOG=debug,my_crate=trace cargo run
```

关键点：

- `tracing` 负责产生事件。
- `tracing-subscriber` 负责处理和输出事件。
- `EnvFilter` 可以通过 `RUST_LOG` 控制日志级别。

## 常见组合

### CLI 工具

```toml
[dependencies]
anyhow = "1"
clap = { version = "4", features = ["derive"] }
tracing = "0.1"
tracing-subscriber = "0.3"
```

常见结构是 `clap` 解析参数，`anyhow` 处理错误，`tracing` 输出日志。

### HTTP 客户端

```toml
[dependencies]
anyhow = "1"
reqwest = { version = "0.12", features = ["json"] }
serde = { version = "1", features = ["derive"] }
serde_json = "1"
tokio = { version = "1", features = ["full"] }
```

常见结构是 `reqwest` 发请求，`serde` 定义请求和响应结构，`anyhow` 简化错误传播。

### 配置文件

```toml
[dependencies]
anyhow = "1"
serde = { version = "1", features = ["derive"] }
toml = "0.8"
```

常见结构是把配置文件读成字符串，再用 `toml::from_str` 转成结构体。
